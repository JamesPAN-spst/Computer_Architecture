多核（Multi-core）与一致性（Coherence）

知识点 1 · SMP 与 NUMA 架构（SMP vs NUMA Architecture）

🎯 为什么研究这个问题

单核处理器的性能提升受到"功耗墙"（Power Wall）和"指令级并行度墙"（ILP Wall）的限制。从 2005 年左右开始，处理器厂商不再追求更高的单核频率，转而在单芯片上集成多个核心——多核时代由此开启。但多核设计面临一个根本性的架构选择：多个核心如何共享内存？

不同的内存共享方式决定了编程模型的复杂度、系统的可扩展性（核心数从 2 到 100+）以及实际的性能表现。SMP 和 NUMA 是两种最重要的共享内存架构，理解它们的区别对于编写高性能并行程序至关重要。

🔍 抽象化方法

SMP（Symmetric Multiprocessing，对称多处理）：所有处理器通过一条共享总线（或交叉开关）连接到统一的主存。每个处理器访问任何内存地址的延迟相同——称为UMA（Uniform Memory Access）。优点是编程简单（不需要考虑数据放置），缺点是共享总线带宽有限，随着核心数增加成为严重瓶颈。典型的 SMP 系统核心数通常不超过 8–16 个。

NUMA（Non-Uniform Memory Access，非均匀内存访问）：每个处理器（或处理器组）有自己的本地内存，通过高速互连网络（如 Intel QPI/UPI、AMD Infinity Fabric、ARM CCIX）也能访问其他处理器的内存。访问本地内存快（~100ns），访问远程内存慢（~150-300ns）。NUMA 避免了共享总线瓶颈，可扩展到数十甚至数百个核心。但程序员或操作系统必须注意数据亲和性（Data Affinity）——尽量让线程访问本地内存。

现代服务器普遍采用 NUMA 架构。即使是消费级的 AMD Ryzen（chiplet 设计）也具有 NUMA 特征——不同 CCD（Core Complex Die）之间的内存访问延迟略有差异。

📐 公式推导 / 模型建立

NUMA 系统的平均内存访问时间取决于本地/远程访问比例：

\[ \text{AMAT}_{\text{NUMA}} = f_{\text{local}} \times t_{\text{local}} + (1 - f_{\text{local}}) \times t_{\text{remote}} \]

其中 \(f_{\text{local}}\) 是本地访问比例，\(t_{\text{local}}\) 和 \(t_{\text{remote}}\) 分别是本地和远程访问延迟。

定义 NUMA 比率（NUMA Ratio）为远程与本地延迟之比：

\[ R_{\text{NUMA}} = \frac{t_{\text{remote}}}{t_{\text{local}}} \]

典型值：Intel 双路服务器 \(R_{\text{NUMA}} \approx 1.5\text{–}2.0\)；四路服务器 \(R_{\text{NUMA}} \approx 2.0\text{–}3.0\)。

例如 \(t_{\text{local}} = 80\text{ns}\)，\(R_{\text{NUMA}} = 2.0\)，\(f_{\text{local}} = 70\%\)：

\[ \text{AMAT} = 0.70 \times 80 + 0.30 \times 160 = 56 + 48 = 104\text{ ns} \]

如果优化数据放置使 \(f_{\text{local}} = 95\%\)：

\[ \text{AMAT} = 0.95 \times 80 + 0.05 \times 160 = 76 + 8 = 84\text{ ns} \]

数据亲和性优化将平均延迟降低了约 19%！

⚠️ 共享总线的 SMP 带宽公式为 \(B_{\text{total}} = B_{\text{bus}}\)（固定的总线带宽），与核心数无关。当核心数 \(n\) 增加时，每核可用带宽为 \(B_{\text{bus}} / n\)，线性下降。这就是 SMP 不可扩展的根本原因。NUMA 的互连网络带宽随节点数增加而扩展。
🌐 实际应用

操作系统 NUMA 感知调度：Linux 内核的 NUMA balancing 机制会自动监测线程的内存访问模式，将频繁访问远程内存的线程迁移到数据所在的 NUMA 节点。可以使用 numactl 工具手动控制线程和内存的 NUMA 亲和性。

数据库优化：大型数据库（如 Oracle、PostgreSQL）的 buffer pool 设计需要考虑 NUMA 拓扑。将 buffer pool 分片到各 NUMA 节点的本地内存上，避免跨节点访问，可以显著提升 TPS（每秒事务数）。

AMD chiplet 架构：AMD Zen 系列使用 chiplet 设计（多个 CCD 通过 Infinity Fabric 互连），形成了芯片内部的 NUMA 结构。理解这一点对于优化游戏和高性能计算的线程调度很重要。

✏️ 例题分析与易错点

题目：一个 4 节点 NUMA 系统，本地内存延迟 100ns，1 跳远程延迟 170ns，2 跳远程延迟 250ns。节点 0 运行一个线程，70% 访问本地，20% 访问 1 跳远程，10% 访问 2 跳远程。计算平均内存延迟。

解答：

\[\text{AMAT} = 0.70 \times 100 + 0.20 \times 170 + 0.10 \times 250 = 70 + 34 + 25 = 129\text{ ns}\]

❌ 常见错误：把所有远程访问都按最远距离计算，得到 \(0.30 \times 250 = 75\) → AMAT = 145ns。实际上不同距离的远程节点延迟不同。

✅ 正确做法：根据拓扑结构分别计算不同距离的远程访问延迟，加权求和。

知识点 2 · MESI 协议（MESI Protocol）

🎯 为什么研究这个问题

在多核系统中，每个核心有自己的私有缓存（L1/L2）。当多个核心缓存了同一个内存地址的数据副本时，一个核心的写操作会使其他核心的副本过时（Stale）。如果不加以管理，一个线程读到的可能是另一个线程已经修改前的旧值——这种数据不一致会导致程序产生不可预测的错误结果。

缓存一致性协议（Cache Coherence Protocol）的任务就是保证：任何核心读到的某地址的值，必须是该地址的最新写入值。MESI 是最经典也是应用最广泛的写失效（Write-Invalidate）一致性协议，被 Intel x86、许多 ARM 实现等广泛采用。

🔍 抽象化方法

MESI 协议为每个缓存行维护四个状态：

M（Modified）：本缓存行已被修改，与主存不一致。只有本核心拥有副本。如果其他核心要读该地址，本核心必须先将数据写回主存（或直接转发）再降为 Shared。

E（Exclusive）：本缓存行未被修改，与主存一致，且只有本核心拥有副本。可以直接转为 Modified（无需通知其他核心），因为没有其他副本。

S（Shared）：本缓存行与主存一致，但可能有其他核心也持有副本。写入前必须先发 Invalidate 消息使其他副本失效。

I（Invalid）：本缓存行无效（空的或已被失效）。访问该地址需要从主存或其他核心获取。

📐 公式推导 / 模型建立

MESI 的关键状态转移（以核心 A 的视角）：

读未命中（Read Miss）：

I → S（如果其他核心持有该行的 S 或 E 副本）或 I → E（如果没有其他核心持有该行）。

写命中（Write Hit）：

E → M（本核心独占且干净 → 直接变脏，无需总线事务）。

S → M（先广播 Invalidate → 其他核心副本变 I → 本核心变 M）。

写未命中（Write Miss）：

I → M（发起 BusRdX 事务：读取数据 + 使其他副本失效，一步完成）。

一致性的总线流量开销分析：设有 \(n\) 个核心，数据块的共享度为 \(s\)（即平均有 \(s\) 个核心持有同一缓存行的副本），则每次写操作产生的失效消息数为：

\[ \text{Invalidations per write} = s - 1 \]

总线流量 = 写操作频率 × 失效消息数。降低共享度（减少不必要的数据共享）是减少一致性流量的关键。

⚠️ E 状态是 MESI 相对于 MSI 协议的关键优化：在 MSI 中，独占访问的缓存行也处于 S 状态，写入时仍需发 Invalidate。而 MESI 中 E 状态表示"独占且干净"，写入时可以静默升级为 M，不需要任何总线事务——这在单线程访问的数据块上节省了大量总线带宽。
🌐 实际应用

MOESI 和 MESIF 变体：AMD 使用 MOESI 协议，增加了 O（Owned）状态，允许修改过的数据直接从一个核心转发到另一个核心，无需先写回主存。Intel 使用 MESIF 协议，增加了 F（Forward）状态来优化多播读请求。

目录协议（Directory Protocol）：在大规模 NUMA 系统中（几十到几百个核心），基于总线的嗅探（Snooping）协议不可扩展——每个核心都要监听每个总线事务。目录协议使用一个集中的目录记录每个缓存行的状态和持有者，只向相关核心发送一致性消息，显著减少了网络流量。

性能工具：Linux 的 perf 工具可以监测缓存一致性相关的硬件计数器，如远程缓存命中次数、失效消息数量等。Intel VTune 的 Memory Access 分析可以精确定位跨核心数据共享导致的性能瓶颈。

✏️ 例题分析与易错点

题目：双核系统中，初始时缓存行 X 不在任何缓存中。按以下顺序执行操作，列出每步后两个核心中该缓存行的 MESI 状态：① Core0 读 X ② Core1 读 X ③ Core0 写 X ④ Core1 读 X

解答：

① Core0 读 X：Core0 发 BusRd，无人持有 → Core0=E，Core1=I。

② Core1 读 X：Core1 发 BusRd，Core0 嗅探到 → Core0 从 E 降为 S，Core1=S。

③ Core0 写 X：Core0 发 Invalidate → Core1 变为 I，Core0 变为 M。

④ Core1 读 X：Core1 发 BusRd，Core0 嗅探到自己是 M → Core0 将数据写回（或直接转发）→ Core0=S，Core1=S。

❌ 常见错误：步骤 ① 中将状态设为 S 而非 E——只有一个核心持有时应为 Exclusive，不是 Shared。

✅ 关键区分：E 状态的条件是"本核心是唯一持有者且数据干净"。

知识点 3 · 伪共享（False Sharing）的性能陷阱

🎯 为什么研究这个问题

考虑以下场景：线程 A 频繁写入变量 counter_a，线程 B 频繁写入变量 counter_b。两个变量在逻辑上完全独立，没有任何数据依赖。程序员理所当然地认为两个线程可以并行运行而互不干扰。

但如果 counter_a 和 counter_b 在内存中相邻（例如在同一个结构体的连续字段中），它们极有可能位于同一缓存行（通常 64 字节）。由于缓存一致性协议以缓存行为粒度工作，线程 A 写 counter_a 会导致线程 B 的缓存行失效，线程 B 写 counter_b 又导致线程 A 的缓存行失效——两个线程不断互相使对方的缓存失效，性能可能下降 10 倍甚至更多！

🔍 抽象化方法

伪共享（False Sharing）的定义：多个处理器核心分别访问不同的数据，但这些数据恰好位于同一缓存行中，导致不必要的缓存一致性通信。称为"伪"共享是因为数据在逻辑上不共享，冲突仅由物理布局引起。

伪共享的识别特征：① 多线程程序在增加核心数时性能不升反降；② perf 显示大量的缓存失效和一致性流量；③ 不同线程写入的变量在内存中距离小于缓存行大小（64 字节）。

伪共享的解决方案核心思想是对齐和填充（Alignment and Padding）：确保不同线程写入的数据位于不同的缓存行中。常见方法包括：① 在变量之间插入填充字节；② 使用编译器对齐属性（如 C++ 的 alignas(64)）；③ 使用语言提供的特殊注解（如 Java 的 @Contended）。

📐 公式推导 / 模型建立

伪共享的性能影响可以用以下模型估算。设缓存行大小为 \(L\) 字节，两个线程分别以频率 \(f_A\) 和 \(f_B\) 写入同一缓存行中的不同位置：

\[ \text{失效频率} = f_A + f_B \quad \text{（每次写都可能使对方失效）} \]

每次失效的代价约为 \(t_{\text{coherence}}\)（一致性协议的延迟，通常 50–200 周期），则额外延迟为：

\[ \text{额外开销} = (f_A + f_B) \times t_{\text{coherence}} \]

对比无伪共享的情况：每个变量在独立的缓存行中，写操作只命中本地缓存（~1-4 周期），额外开销几乎为零。

使用填充解决伪共享后，内存开销增加：每个独占变量需要占据一个完整缓存行（64 字节），即使变量本身只有 4 或 8 字节。对于少量热点变量，这个空间代价完全值得。

⚠️ 伪共享只在多核写场景下有严重影响。如果多个核心只是读同一缓存行（例如共享只读配置），则缓存行在所有核心中都处于 Shared 状态，不会产生失效——这是真正的共享读，完全没有问题。
🌐 实际应用

C/C++ 中的填充技术：

使用 alignas(64)：struct alignas(64) PaddedCounter { int64_t value; };。或者手动填充：struct PaddedCounter { int64_t value; char padding[56]; };（假设 int64_t 为 8 字节，填充到 64 字节）。

Java 的 @Contended 注解：Java 8 引入了 @sun.misc.Contended（Java 9+ 为 @jdk.internal.vm.annotation.Contended），JVM 会自动在标注字段周围添加 128 字节的填充。Java 的 LongAdder 内部就使用了此技术来避免多线程计数时的伪共享。

Linux 内核：Linux 内核中大量使用 ____cacheline_aligned_in_smp 宏来对齐关键的每 CPU 变量（per-CPU variables），避免不同 CPU 频繁写入的变量落在同一缓存行中。struct rq（运行队列）就是一个典型例子。

✏️ 例题分析与易错点

题目：以下结构体在 4 核系统中被 4 个线程分别写入 count[0] 到 count[3]，是否存在伪共享？如何解决？

struct Counters { int64_t count[4]; }; // 4×8 = 32 字节

解答：是的，存在伪共享。4 个 int64_t 共 32 字节，全部位于同一个 64 字节缓存行中。4 个线程的写操作会互相使对方的缓存失效。

解决方案：struct PaddedCounters { alignas(64) int64_t count[4][8]; };（每个计数器占一个完整缓存行），或使用结构体数组 + 对齐。

❌ 常见错误：认为不同数组元素自动在不同缓存行——实际上连续的小元素很可能在同一缓存行。

✅ 正确做法：对于多线程写入的热点变量，必须显式填充到缓存行边界。

知识点 4 · 内存模型（Memory Model）与内存屏障（Memory Barrier）

🎯 为什么研究这个问题

缓存一致性（Coherence）保证了所有核心最终能看到对同一地址的最新写入。但它不保证不同地址之间的写入顺序！考虑以下经典例子：

线程 A：data = 42; flag = 1;
线程 B：while (flag == 0); print(data);

程序员期望：线程 B 看到 flag == 1 时，data 已经是 42。但现代处理器和编译器可能对指令进行重排序（Reordering）——线程 A 的 Store 操作可能以不同于程序顺序的方式对线程 B 可见。在某些处理器上，线程 B 可能看到 flag == 1 但 data 仍然是旧值！

内存一致性模型（Memory Consistency Model）定义了多线程程序中内存操作的可见顺序规则。它告诉程序员："在什么条件下，一个核心的写操作一定对另一个核心的读操作可见？"

🔍 抽象化方法

内存模型按强弱排列：

顺序一致性（Sequential Consistency, SC）：最强最直觉的模型。所有核心的内存操作仿佛在一个全局序列中执行，每个核心的操作顺序与程序顺序一致。缺点：禁止了大部分硬件优化（如 Store Buffer），性能差。

总存储有序（Total Store Order, TSO）：x86 架构采用。允许 Store-Load 重排序（一个 Store 后面紧跟的 Load 可以先执行），但保持 Store-Store、Load-Load、Load-Store 的顺序。TSO 下上面的 flag 例子是安全的（Store-Store 不重排）。

松散模型（Relaxed / Weak Order）：ARM、RISC-V、PowerPC 采用。几乎所有类型的重排序都被允许——编译器和处理器可以自由重排 Load 和 Store，除非显式插入内存屏障。性能最好，但编程最复杂。

内存屏障（Memory Barrier / Fence）：一种特殊指令，禁止屏障前后的内存操作被重排。不同类型的屏障约束不同类型的重排序：Store Fence（阻止 Store-Store 重排）、Load Fence（阻止 Load-Load 重排）、Full Fence（阻止所有重排）。

📐 公式推导 / 模型建立

重排序的类型矩阵（✓ = 允许重排，✗ = 禁止重排）：

x86 (TSO)：

\[ \begin{array}{c|cc} & \text{后续 Load} & \text{后续 Store} \\ \hline \text{前序 Load} & \times & \times \\ \text{前序 Store} & \checkmark & \times \end{array} \]

ARM/RISC-V (Relaxed)：

\[ \begin{array}{c|cc} & \text{后续 Load} & \text{后续 Store} \\ \hline \text{前序 Load} & \checkmark & \checkmark \\ \text{前序 Store} & \checkmark & \checkmark \end{array} \]

x86 的 TSO 模型只允许一种重排（Store→Load），因此大部分多线程程序不需要显式屏障。而 ARM 的松散模型允许所有四种重排，程序员必须在关键位置插入屏障指令（如 DMB、DSB）。

内存屏障的性能代价：一条 Full Fence 指令（如 x86 的 MFENCE）通常需要刷空 Store Buffer，代价约 20–100 个周期。因此应该尽量减少屏障的使用——只在确实需要顺序保证的地方使用。

⚠️ 编译器也会重排内存操作！即使硬件保证了顺序，如果编译器将 data = 42 和 flag = 1 重排了，结果同样不正确。C/C++ 的 volatile 阻止编译器重排但不阻止硬件重排。必须使用 std::atomic 或编译器屏障（如 asm volatile("" ::: "memory")）同时约束两者。
🌐 实际应用

C++11 内存序（Memory Order）：C++11 定义了六种内存序：relaxed、consume、acquire、release、acq_rel、seq_cst。其中 acquire 和 release 是最常用的一对：release 保证之前的所有写操作在 store 之前完成（Store Fence）；acquire 保证之后的所有读操作在 load 之后进行（Load Fence）。配对使用可以实现高效的发布-订阅模式。

无锁数据结构（Lock-Free Data Structures）：无锁队列、无锁栈等高性能并发数据结构大量使用 acquire/release 语义来保证正确的发布顺序，避免使用重量级的互斥锁（mutex）。例如 Michael-Scott 无锁队列，每个 CAS（Compare-And-Swap）操作使用 acq_rel 语义。

Linux 内核的屏障原语：Linux 内核定义了 smp_rmb()（读屏障）、smp_wmb()（写屏障）、smp_mb()（全屏障）等宏。在 x86 上，smp_wmb() 只是编译器屏障（因为 TSO 已保证 Store-Store 顺序），而在 ARM 上则会生成 DMB ISHST 指令。

✏️ 例题分析与易错点

题目：在 ARM（松散模型）上执行以下代码，不加屏障是否安全？

线程 A：data = 42; flag = 1;
线程 B：while (flag == 0); r = data;

解答：不安全！ARM 的松散模型允许 Store-Store 重排：线程 A 的 flag = 1 可能先于 data = 42 对线程 B 可见。也允许 Load-Load 重排：线程 B 的 r = data 可能先于 flag 的读取执行（乱序推测执行）。

修复方案：线程 A 在两个 Store 之间插入写屏障（DMB ISHST 或使用 release 语义写 flag）；线程 B 在读 flag 和读 data 之间插入读屏障（DMB ISHLD 或使用 acquire 语义读 flag）。

❌ 常见错误：在 x86 上测试通过就认为代码正确——x86 的 TSO 模型隐藏了 Store-Store 重排问题，移植到 ARM 后可能出错。

✅ 正确做法：使用 C++11 std::atomic 并指定合适的 memory order，让编译器为不同平台生成正确的屏障指令。
