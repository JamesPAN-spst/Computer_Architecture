Chapitre 1

冯·诺依曼架构

知识点 1 · 存储程序（Stored Program）思想的革命

🎯 为什么研究这个问题

在冯·诺依曼之前，计算机如 ENIAC 需要通过重新连接电缆和开关来"编程"。每次换一个计算任务，工程师可能要花数天时间重新配线。这意味着硬件和软件无法分离——机器的"行为"被物理连线所决定。

存储程序思想的提出，是计算机历史上最关键的抽象突破之一。它让我们可以在不改变硬件的前提下，仅通过改变存储器中的内容就能让同一台机器完成截然不同的任务——从科学计算到文字处理，从图像渲染到网络通信。

理解这一思想，是理解现代通用计算机本质的起点。

🔍 抽象化方法

冯·诺依曼模型将计算机抽象为五大组件：运算器（ALU）、控制器（Control Unit）、存储器（Memory）、输入设备和输出设备。其中最革命性的一点是：指令和数据采用相同的编码方式存储在同一个存储器中。

这种统一存储的抽象带来了深刻的结果：程序本身变成了可操作的数据。编译器可以生成机器码并写入存储器；操作系统可以加载、切换、终止不同的程序；自修改代码（self-modifying code）甚至可以在运行时改变自身的指令。

与之对比的是哈佛架构（Harvard Architecture），它将指令存储器和数据存储器物理分离，使得两者可以同时访问，消除了总线带宽瓶颈。现代处理器的 L1 缓存往往采用哈佛结构（分为 I-Cache 和 D-Cache），但在主存层面仍然是冯·诺依曼的统一存储。

📐 公式推导 / 模型建立

冯·诺依曼的执行模型可以用一个简单的循环来描述：

\[ \textbf{while}\ (\text{halt} = 0)\ \{\ \text{IR} \leftarrow \text{Mem}[\text{PC}];\ \text{PC} \leftarrow \text{PC} + 4;\ \text{Execute}(\text{IR});\ \} \]

其中 \(\text{PC}\)（Program Counter）指向当前要执行的指令地址，\(\text{IR}\)（Instruction Register）保存取出的指令。每次循环：先取指、再更新 PC、然后执行。这就是取指-执行循环（Fetch-Execute Cycle）。

假设存储器带宽为 \(B\) bytes/s，每条指令平均 \(L\) 字节，平均需要搬运 \(D\) 字节数据，则处理器执行速率受限于：

\[ \text{IPS}_{\max} = \frac{B}{L + D} \]

这就是冯·诺依曼瓶颈的量化表达——所有指令取指和数据搬运共享同一条总线，总线带宽成为性能天花板。

⚠️ 冯·诺依曼瓶颈不是说 CPU 不够快，而是指 CPU 和存储器之间的数据搬运速率跟不上 CPU 处理速度。现代体系结构通过多级缓存、预取、DMA 等技术来缓解这一瓶颈，但未从根本上消除。
🌐 实际应用

操作系统的多任务：正是因为程序存储在可读写的存储器中，操作系统才能将多个程序加载到内存的不同位置，通过切换 PC 的值来交替执行不同的程序，实现多任务。

JIT 编译器：Java 虚拟机和 JavaScript 引擎中的即时编译器（JIT）在运行时将字节码编译为本地机器码，写入内存后直接跳转执行——这就是"指令即数据"的最佳体现。

嵌入式系统的哈佛变体：许多微控制器（如 AVR、PIC）使用修改版的哈佛架构，将程序存储在 Flash 中，数据存储在 SRAM 中。但编程模型仍然保持冯·诺依曼的顺序执行语义。

✏️ 例题分析与易错点

题目：一台冯·诺依曼结构计算机，存储器带宽为 8 GB/s，指令平均长度 4 字节，每条指令平均需搬运 4 字节数据。求理论最大指令执行速率。

解答： \(\text{IPS}_{\max} = \frac{8 \times 10^9}{4 + 4} = 1 \times 10^9 = 1\text{ GIPS}\)

❌ 常见错误：忽略数据搬运的带宽消耗，只用 \(B / L\) 计算，高估了实际性能。

✅ 正确做法：必须同时考虑指令取指和数据传输对总线带宽的占用。

知识点 2 · 单周期 CPU 数据通路

🎯 为什么研究这个问题

冯·诺依曼模型只告诉我们"取指-执行"的抽象过程，但没有告诉我们硬件层面到底如何实现。数据通路（Datapath）就是这个抽象的硬件实现——它定义了指令在 CPU 内部的物理路径：从哪里取操作数、送到哪里运算、结果写到哪里。

理解数据通路是理解 CPU 设计的基础。无论是后续的流水线优化、乱序执行还是超标量设计，都是在单周期数据通路的基础上进行改进。

🔍 抽象化方法

单周期数据通路的核心组件包括：

① 程序计数器（PC）：指向下一条要执行的指令地址。

② 指令存储器（Instruction Memory）：根据 PC 输出当前指令。

③ 寄存器文件（Register File）：提供两个读端口和一个写端口，支持同时读取两个源寄存器并写入一个目标寄存器。

④ ALU：根据控制信号执行加、减、与、或等运算。

⑤ 数据存储器（Data Memory）：执行 load/store 时访问。

⑥ 多路选择器（MUX）：根据控制信号选择不同的数据源，如选择寄存器值还是立即数送入 ALU。

📐 公式推导 / 模型建立

对于一条 R 型指令（如 add rd, rs1, rs2），数据通路的执行过程可以形式化为：

\[ \text{Reg}[rd] \leftarrow \text{Reg}[rs1] \mathbin{\oplus_{\text{ALU}}} \text{Reg}[rs2] \]

对于 load 指令（如 lw rd, offset(rs1)）：

\[ \text{Reg}[rd] \leftarrow \text{Mem}\bigl[\text{Reg}[rs1] + \text{SignExt}(\text{offset})\bigr] \]

单周期时钟周期的确定取决于最长路径。假设各组件延迟为：取指 \(t_{IF}\)、寄存器读 \(t_{RF}\)、ALU \(t_{ALU}\)、存储器访问 \(t_{MEM}\)、寄存器写 \(t_{WB}\)，则时钟周期为：

\[ T_{clk} \geq t_{IF} + t_{RF} + t_{ALU} + t_{MEM} + t_{WB} \]
⚠️ 单周期设计中，即使最简单的 R 型指令不需要访存，其时钟周期也必须等于最慢的 load 指令的延迟。这种"木桶效应"是单周期 CPU 效率低下的根本原因。
🌐 实际应用

教学处理器：单周期 CPU 虽然在实际产品中几乎不使用，但它是计算机组成原理课程的核心教学模型。RISC-V 和 MIPS 的教学版处理器通常从单周期开始设计。

FPGA 原型验证：在 FPGA 上实现单周期 CPU 是验证指令集正确性的第一步。由于单周期设计逻辑简单、状态少，调试起来比流水线容易得多。

理解关键路径：分析单周期数据通路的延迟，能帮助我们理解为什么后续要引入流水线——将长路径切分成多个短阶段，每个阶段的延迟更短，时钟频率因此可以大幅提升。

✏️ 例题分析与易错点

题目：假设取指 200ps、寄存器读 100ps、ALU 200ps、存储器访问 200ps、寄存器写 100ps。单周期 CPU 的最高时钟频率是多少？

解答：最慢路径为 load 指令：\(T = 200 + 100 + 200 + 200 + 100 = 800\text{ps}\)。最高频率 \(f = 1/800\text{ps} = 1.25\text{ GHz}\)。

❌ 常见错误：只算 R 型指令的路径（省略 MEM 阶段），得到 600ps、1.67 GHz。这是错误的，因为单周期 CPU 不能为不同指令分配不同周期。

✅ 正确做法：总是以最慢的指令路径确定时钟周期。

知识点 3 · 取指-译码-执行-访存-写回（IF/ID/EX/MEM/WB）

🎯 为什么研究这个问题

理解了单周期数据通路之后，一个自然的问题是：能不能把一条指令的执行拆分成多个步骤，每个步骤在不同的硬件模块上完成？这样不仅能让不同指令使用不同数量的周期（多周期设计），还为后续的流水线打下基础。

五级划分（IF/ID/EX/MEM/WB）是 MIPS 和 RISC-V 经典教学模型的基础，也是理解现代处理器微架构的起点。即使现代处理器的流水线深度可能是 10-20 级甚至更多，核心功能仍然可以映射到这五个逻辑阶段。

🔍 抽象化方法

五个阶段的职责划分如下：

IF（Instruction Fetch）：用 PC 的值从指令存储器中取出指令，同时计算 \(\text{PC} + 4\) 作为下一条指令的默认地址。

ID（Instruction Decode / Register Read）：解析指令字段（opcode、rs1、rs2、rd、funct、immediate），同时从寄存器文件中读取源操作数。控制器在此阶段生成控制信号。

EX（Execute / Address Calculation）：ALU 执行运算（算术逻辑运算或地址计算）。对于分支指令，ALU 判断分支条件是否成立。

MEM（Memory Access）：load 指令读取数据存储器，store 指令写入数据存储器。其他指令在此阶段"空过"。

WB（Write Back）：将结果（ALU 输出或内存读取值）写回寄存器文件中的目标寄存器。

📐 公式推导 / 模型建立

在多周期设计中，不同类型的指令使用不同数量的周期。设每阶段耗时为 \(t_s\)，各类指令的周期数为：

\[ \begin{aligned} \text{R-type} &: \text{IF} + \text{ID} + \text{EX} + \text{WB} = 4\,t_s \\ \text{Load} &: \text{IF} + \text{ID} + \text{EX} + \text{MEM} + \text{WB} = 5\,t_s \\ \text{Store} &: \text{IF} + \text{ID} + \text{EX} + \text{MEM} = 4\,t_s \\ \text{Branch} &: \text{IF} + \text{ID} + \text{EX} = 3\,t_s \end{aligned} \]

多周期设计的平均 CPI（Cycles Per Instruction）取决于指令混合比例。若 R-type 占 40%、Load 占 20%、Store 占 10%、Branch 占 30%：

\[ \text{CPI}_{\text{avg}} = 0.4 \times 4 + 0.2 \times 5 + 0.1 \times 4 + 0.3 \times 3 = 3.9 \]
⚠️ 多周期设计虽然提高了时钟频率（每阶段更短），但 CPI > 1。只有流水线才能同时做到高时钟频率和接近 1 的 CPI。切勿将多周期与流水线混淆！
🌐 实际应用

RISC-V 教学处理器：许多 RISC-V 教材（如 Patterson & Hennessy 的《计算机组成与设计》）以这五个阶段为框架，逐步构建从单周期到流水线的处理器。

现代处理器的超深流水线：Intel Pentium 4（NetBurst 架构）的流水线深达 20+ 级，本质上是将经典五阶段进一步细分。例如 IF 被拆成多个子阶段（预取、对齐、解码预处理等）。

微操作（μop）分解：x86 处理器将复杂的 CISC 指令在 ID 阶段分解为多个 RISC 风格的微操作，后续阶段处理这些微操作而非原始指令。

✏️ 例题分析与易错点

题目：一条 lw x5, 8(x10) 指令依次经历哪些阶段？每个阶段做什么？

解答：

① IF：从 Mem[PC] 取出指令 → IR。

② ID：解析得到 rs1=x10, rd=x5, offset=8；读取 Reg[x10]。

③ EX：ALU 计算地址 = Reg[x10] + 8。

④ MEM：用计算得到的地址访问数据存储器，读取 Mem[Reg[x10]+8]。

⑤ WB：将读取的值写入 Reg[x5]。

❌ 常见错误：认为 store 指令也有 WB 阶段——store 不需要写回寄存器，它在 MEM 阶段完成写入存储器后即结束。

知识点 4 · 控制器（Control Unit）的设计

🎯 为什么研究这个问题

数据通路就像一座城市的道路网络，而控制器是交通信号灯和调度中心。没有控制器，数据就不知道走哪条路、在哪里转弯。控制器通过分析指令中的 opcode 和 funct 字段，生成一组控制信号来指挥数据通路中各个组件的行为。

一个错误的控制信号就可能让 ALU 执行错误运算、将结果写入错误的寄存器、或在不该访存时访问存储器。因此，控制器的正确设计是 CPU 功能正确性的关键保障。

🔍 抽象化方法

控制器的设计有两种经典方法：

① 硬布线控制（Hardwired Control）：使用组合逻辑电路（真值表 → 逻辑门）直接从 opcode 生成控制信号。速度快但修改困难——每次指令集变化都要重新设计电路。

② 微程序控制（Microprogrammed Control）：将每条机器指令的执行步骤编码为微指令序列，存储在控制存储器（ROM）中。执行机器指令时，控制器从 ROM 中读取微指令并执行。灵活性强，修改指令行为只需修改微程序。

典型的控制信号包括：RegWrite（寄存器写使能）、MemRead/MemWrite（存储器读/写）、ALUSrc（ALU 第二操作数来源）、MemtoReg（写回值来源）、Branch（分支使能）等。

📐 公式推导 / 模型建立

以 RISC-V 为例，硬布线控制器的真值表可以形式化为布尔函数。假设 opcode 为 7 位，则控制信号是 opcode 的函数：

\[ \text{RegWrite} = f_1(\text{opcode}), \quad \text{ALUSrc} = f_2(\text{opcode}), \quad \text{MemRead} = f_3(\text{opcode}), \ldots \]

例如对于 RISC-V 的三种典型指令：

\[ \begin{array}{|c|c|c|c|c|c|} \hline \textbf{指令类型} & \text{RegWrite} & \text{ALUSrc} & \text{MemRead} & \text{MemWrite} & \text{Branch} \\ \hline \text{R-type} & 1 & 0 & 0 & 0 & 0 \\ \text{lw} & 1 & 1 & 1 & 0 & 0 \\ \text{sw} & 0 & 1 & 0 & 1 & 0 \\ \text{beq} & 0 & 0 & 0 & 0 & 1 \\ \hline \end{array} \]
⚠️ ALUOp 信号通常需要二级解码：先由主控制器根据 opcode 生成 ALUOp，再由 ALU 控制器结合 ALUOp 和 funct 字段生成最终的 ALU 运算选择信号。很多学生在考试中遗漏这一层级，导致对控制信号的理解不完整。
🌐 实际应用

x86 的微程序传统：Intel 从 386 时代起大量使用微程序控制。每条复杂的 CISC 指令被翻译为一系列微操作（μops），通过微程序序列器驱动执行。这也是 x86 能在保持向后兼容的同时不断添加新指令的秘密。

RISC 的硬布线哲学：RISC 架构（ARM、RISC-V、MIPS）的指令格式规整、指令种类少，非常适合硬布线控制。这带来了更快的译码速度和更低的功耗。

可重配置处理器：某些 FPGA 软核处理器（如 NIOS II）使用微程序控制，用户可以通过修改微程序来自定义指令行为，实现特定领域的加速指令。

✏️ 例题分析与易错点

题目：在执行 sw x5, 12(x10)（store word）时，以下控制信号哪个为 1？(A) RegWrite (B) MemWrite (C) MemtoReg (D) Branch

解答：选 (B)。sw 需要写入数据存储器，所以 MemWrite = 1。sw 不写寄存器（RegWrite = 0），MemtoReg 无关紧要（因为不写寄存器），也不是分支指令（Branch = 0）。

❌ 常见错误：混淆 load 和 store 的控制信号——load 需要 MemRead = 1 且 RegWrite = 1，而 store 只需要 MemWrite = 1。

✅ 记忆口诀：Load 读存储器写寄存器，Store 读寄存器写存储器。
