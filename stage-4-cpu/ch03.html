Chapitre 3

缓存（Cache）

知识点 1 · 局部性原理（Principle of Locality）

🎯 为什么研究这个问题

处理器速度和主存速度的差距在过去几十年中持续扩大。现代 CPU 一个时钟周期不到 1ns，而 DRAM 的访问延迟约 50-100ns，差距超过 100 倍。如果每次 CPU 需要数据都要等待主存响应，绝大部分时间都浪费在等待上。

缓存（Cache）之所以有效，不是因为技术上的奇迹，而是因为程序行为的一个基本特征——局部性（Locality）。只要程序对数据的访问具有局部性，一块小而快的缓存就能拦截大部分访问请求，避免频繁访问慢速主存。

🔍 抽象化方法

局部性分为两种：

时间局部性（Temporal Locality）：如果一个数据项刚被访问过，它很可能在近期再次被访问。例如：循环变量 i、频繁调用的函数代码、全局配置数据等。

空间局部性（Spatial Locality）：如果一个地址被访问，附近地址的数据很可能也会被访问。例如：数组的顺序遍历、结构体中连续字段的访问、代码段的顺序执行等。

缓存设计就是针对这两种局部性进行优化：时间局部性决定了哪些数据应该被保留（替换策略）；空间局部性决定了每次从主存取多大块数据（缓存行大小）。

📐 公式推导 / 模型建立

缓存的有效性可以用平均访存时间（AMAT）来量化：

\[ \text{AMAT} = t_{\text{hit}} + r_{\text{miss}} \times t_{\text{penalty}} \]

其中 \(t_{\text{hit}}\) 是缓存命中时间，\(r_{\text{miss}}\) 是未命中率，\(t_{\text{penalty}}\) 是未命中惩罚（从下一级存储取数据的时间）。

对于多级缓存（L1 + L2）：

\[ \text{AMAT} = t_{L1} + r_{L1} \times \bigl(t_{L2} + r_{L2} \times t_{\text{mem}}\bigr) \]

例如 \(t_{L1} = 1\text{ns}\)、\(r_{L1} = 5\%\)、\(t_{L2} = 5\text{ns}\)、\(r_{L2} = 20\%\)、\(t_{\text{mem}} = 100\text{ns}\)：

\[ \text{AMAT} = 1 + 0.05 \times (5 + 0.2 \times 100) = 1 + 0.05 \times 25 = 2.25\text{ ns} \]
⚠️ AMAT 公式中的 \(r_{L2}\) 是局部未命中率（local miss rate），即 L2 未命中次数 / L2 访问次数。而不是全局未命中率。很多考题在此设陷阱，务必区分这两个概念。
🌐 实际应用

多级缓存层次：现代处理器通常有 3 级缓存：L1（~32-64KB, ~1ns）、L2（~256KB-1MB, ~5ns）、L3（~8-64MB, ~15ns），逐级增大但变慢。这种分层设计是对不同粒度局部性的多层次利用。

数据结构对局部性的影响：数组具有优秀的空间局部性（元素在内存中连续存放），而链表的空间局部性很差（节点可能散布在内存各处）。这解释了为什么在缓存系统中数组遍历通常比链表遍历快数倍。

矩阵分块（Tiling）：大矩阵乘法通过将矩阵分成小块，使得每块能完全放入缓存中，从而利用时间局部性避免反复从主存读取，可以将性能提升一个数量级。

✏️ 例题分析与易错点

题目：L1 命中时间 2 周期，L1 未命中率 8%，未命中惩罚 50 周期。计算 AMAT。

解答：\(\text{AMAT} = 2 + 0.08 \times 50 = 2 + 4 = 6\) 周期。

❌ 常见错误：将命中时间和未命中惩罚搞混——命中时间是访问缓存本身的延迟，未命中惩罚是去下一级取数据的额外延迟。

✅ 正确理解：AMAT = 命中时间 + 未命中率 × 未命中惩罚。每次访问都会产生命中时间，只有未命中时才额外付出惩罚。

知识点 2 · 直接映射（Direct Mapped）vs 组相联（Set-Associative）

🎯 为什么研究这个问题

缓存的核心问题之一是：主存有数百万个地址，缓存只有几千到几万行，如何决定每个地址应该放在缓存的哪个位置？这个映射策略直接影响命中率、查找速度和硬件复杂度。

不同的映射策略在灵活性和实现复杂度之间做出不同的权衡，理解这些权衡是理解缓存设计的关键。

🔍 抽象化方法

直接映射（Direct Mapped）：每个主存地址只能映射到缓存中的唯一一行。用地址的 Index 字段确定行号：\(\text{行号} = \text{地址} \bmod \text{缓存行数}\)。优点是查找快（只看一个位置），缺点是冲突率高。

全相联（Fully Associative）：任何地址可以放在缓存的任意一行。灵活性最高、冲突最少，但需要同时比较所有行的 Tag，硬件复杂度极高。通常只用于小型缓存（如 TLB）。

N 路组相联（N-way Set-Associative）：折中方案。将缓存分成若干组（set），每组有 N 行。地址先通过 Index 确定组号，再在组内 N 行中查找。直接映射是 1 路组相联；全相联是行数路组相联。

📐 公式推导 / 模型建立

设缓存总大小为 \(C\) 字节，缓存行大小为 \(B\) 字节，相联度为 \(N\) 路，则：

\[ \text{组数} = S = \frac{C}{N \times B} \]

地址划分为三个字段：

\[ \underbrace{\text{Tag}}_{t \text{ bits}} \mid \underbrace{\text{Index}}_{s \text{ bits}} \mid \underbrace{\text{Offset}}_{b \text{ bits}} \]

其中 \(b = \log_2 B\)，\(s = \log_2 S\)，\(t = \text{地址位数} - s - b\)。

例如 32 位地址、16KB 缓存、64B 行、4 路组相联：

\[ S = \frac{16384}{4 \times 64} = 64, \quad b = 6, \quad s = 6, \quad t = 32 - 6 - 6 = 20 \]
⚠️ 增加相联度可以减少冲突未命中，但硬件代价是每次查找需要比较 N 个 Tag。对于 L1 缓存，由于时序要求极其严格，通常选择 4-8 路而非全相联。L3 缓存由于延迟容忍度更高，可以使用 16 路甚至更高的相联度。
🌐 实际应用

Intel Core i7 缓存配置：L1 I-Cache 和 D-Cache 各 32KB、8 路组相联；L2 为 256KB、4 路或 8 路；L3 为 8-30MB、16 路组相联。

TLB（Translation Lookaside Buffer）：页表缓存通常采用全相联设计（因为条目数少，只有 32-64 项），以最大化命中率。

ARM 的 VIVT vs PIPT：ARM 早期处理器使用虚拟地址索引的缓存（VIVT）以缩短访问延迟，但会遇到别名（aliasing）问题。现代 ARM 处理器使用物理地址索引和标签（PIPT）或混合方案（VIPT）。

✏️ 例题分析与易错点

题目：一个 2 路组相联缓存，共 8 个缓存行，行大小 16 字节。地址 0x0000、0x0040、0x0080 映射到哪些组？是否会冲突？

解答：组数 = 8 / 2 = 4 组。Offset = 4 位（16 字节），Index = 2 位（4 组）。0x0000 → Index=00（组 0）；0x0040=64 → Index=00（组 0）；0x0080=128 → Index=00（组 0）。三者都映射到组 0，但组 0 有 2 行，所以 0x0000 和 0x0040 可以共存，0x0080 进入时如果组满则需要替换。

❌ 常见错误：忘记先除去 Offset 位再取 Index，直接用整个地址对组数取模。

✅ 正确做法：先右移 \(b\) 位去掉 Offset，再取低 \(s\) 位作为 Index。

知识点 3 · 未命中的三类（冷启动 / 容量 / 冲突）

🎯 为什么研究这个问题

缓存未命中是影响程序性能的关键因素。但"未命中"并非铁板一块——不同原因导致的未命中需要不同的解决方案。盲目增大缓存可能只能减少容量未命中，对冲突未命中无济于事；增大缓存行只能减少冷启动未命中，反而可能加剧冲突。

3C 分类模型将未命中按原因分为三类，为缓存优化提供了清晰的分析框架。

🔍 抽象化方法

冷启动未命中（Compulsory Miss / Cold Start Miss）：首次访问一个从未出现在缓存中的数据块。无论缓存多大、相联度多高，首次访问都必然未命中。也称"首次引用未命中"。

容量未命中（Capacity Miss）：缓存容量不足以容纳程序的整个工作集（Working Set），导致之前加载过的数据被替换后又被需要。即使使用全相联缓存，如果容量不够仍会发生。

冲突未命中（Conflict Miss）：在非全相联缓存中，多个地址映射到同一组（或同一行），互相竞争有限的位置。如果使用全相联缓存则不会发生冲突未命中。也称"碰撞未命中（Collision Miss）"。

📐 公式推导 / 模型建立

3C 分析方法的实验步骤：

① 用无限大全相联缓存模拟，此时只有冷启动未命中 → 得到 \(M_{\text{compulsory}}\)。

② 用实际大小的全相联缓存模拟 → 总未命中 = \(M_{\text{compulsory}} + M_{\text{capacity}}\)。

③ 用实际大小和实际相联度的缓存模拟 → 总未命中 = \(M_{\text{compulsory}} + M_{\text{capacity}} + M_{\text{conflict}}\)。

\[ M_{\text{total}} = M_{\text{compulsory}} + M_{\text{capacity}} + M_{\text{conflict}} \]

各类未命中的优化方向：

\[ \begin{aligned} M_{\text{compulsory}} &\downarrow \quad \text{增大行大小、预取（Prefetching）} \\ M_{\text{capacity}} &\downarrow \quad \text{增大缓存容量} \\ M_{\text{conflict}} &\downarrow \quad \text{增大相联度、改善地址映射} \end{aligned} \]
⚠️ 增大缓存行虽然减少冷启动未命中，但在缓存总大小不变的情况下会减少行数，反而可能增加冲突未命中。这是一个经典的权衡（trade-off），最优行大小通常通过实验或模拟确定（典型值 32-128 字节）。
🌐 实际应用

硬件预取（Prefetching）：现代处理器通过检测访问模式（如步长固定的数组遍历），提前从主存取数据到缓存中，有效减少冷启动和部分容量未命中。Intel 的 L2 预取器可检测多达 32 种不同的步长模式。

Victim Cache：一种小型全相联缓存，专门存放从 L1 被替换出来的缓存行。当发生冲突未命中时，先查 Victim Cache，如果命中就直接取回，有效减少冲突未命中。

编译器优化：循环分块（Loop Tiling）、数组填充（Array Padding）等编译器技术通过改变数据访问模式来减少容量和冲突未命中。

✏️ 例题分析与易错点

题目：直接映射缓存有 4 行（行大小 1 字），反复访问地址 0、4、8、0、4、8……未命中类型是什么？

解答：地址 0→行 0，地址 4→行 0，地址 8→行 0（假设行大小 1 字，4 行）。三个地址都映射到行 0，互相驱逐，每次访问都未命中。首次 3 个是冷启动未命中，后续全是冲突未命中。

❌ 常见错误：将所有重复未命中归为容量未命中——但这里只有 3 个不同的数据块，缓存有 4 行足够容纳，问题出在映射冲突。

✅ 判断方法：如果改为全相联缓存后未命中消失，就是冲突未命中；如果仍然未命中，才是容量未命中。

知识点 4 · 写策略（Write-Through / Write-Back）

🎯 为什么研究这个问题

读操作很简单——缓存命中就直接返回数据。但写操作要复杂得多：当 CPU 修改了缓存中的数据，什么时候将这个修改反映到下一级存储（L2 或主存）？是立刻同步还是延后批量写回？

写策略直接影响：① 主存和缓存之间的一致性；② 写操作的延迟；③ 总线上的写流量。不同的应用场景需要不同的权衡。

🔍 抽象化方法

写直达（Write-Through）：每次写操作同时更新缓存和下一级存储。优点：缓存和主存始终一致，简化了多核一致性协议。缺点：每次写都产生总线流量，延迟高。通常搭配写缓冲区（Write Buffer）来隐藏写延迟。

写回（Write-Back）：写操作只更新缓存，标记该行为"脏"（Dirty）。当脏行被替换时才写回下一级存储。优点：大幅减少写流量，尤其是对同一数据的多次写只产生一次回写。缺点：实现复杂，替换脏行时有额外延迟。

还需考虑写未命中策略：① 写分配（Write-Allocate）：先将目标块从主存加载到缓存再写。② 写不分配（No-Write-Allocate）：直接写入下一级存储，不影响缓存。通常 Write-Back 搭配 Write-Allocate，Write-Through 搭配 No-Write-Allocate。

📐 公式推导 / 模型建立

设写操作比例为 \(f_w\)，缓存命中率为 \(h\)。对于 Write-Through：

\[ \text{写流量} = f_w \times N_{\text{total}} \quad \text{（每次写都产生总线事务）} \]

对于 Write-Back，写流量取决于脏行被替换的频率 \(r_{\text{dirty}}\)：

\[ \text{写流量} = r_{\text{dirty}} \times (1 - h) \times N_{\text{total}} \quad \text{（仅在脏行被替换时产生）} \]

通常 \(r_{\text{dirty}} \times (1-h) \ll f_w\)，所以 Write-Back 产生的总线写流量远小于 Write-Through。

⚠️ Write-Through 并不意味着每次写都要等主存完成——写缓冲区（Write Buffer）可以将写操作排队异步执行，CPU 无需等待。但如果写缓冲区满了（写速度超过主存吸收能力），CPU 仍然需要 stall。
🌐 实际应用

L1 Cache 的选择：大多数现代处理器的 L1 D-Cache 使用 Write-Back + Write-Allocate 策略，以最小化写流量和延迟。

GPU 缓存：GPU 的 L1 缓存通常使用 Write-Through 或 Write-Evict 策略，因为 GPU 的大量线程使得 Write-Back 的一致性管理过于复杂。

多核一致性：Write-Back 策略下，修改只在 L1 中可见，其他核心不知道数据已被修改。这需要缓存一致性协议（如 MESI）来协调。Write-Through 天然保证下一级存储的一致性，简化了协议。

✏️ 例题分析与易错点

题目：程序执行 1000 条指令，其中 200 次写操作。缓存命中率 95%，Write-Through 和 Write-Back 分别产生多少次主存写操作？（假设 Write-Back 中 50% 的替换行是脏的）

解答：

Write-Through：200 次（每次写都写主存）。

Write-Back：未命中 = 1000 × 5% = 50 次，其中需要替换脏行 = 50 × 50% = 25 次主存写操作。

❌ 常见错误：Write-Back 下把所有写操作都算成主存写——实际上 Write-Back 只在脏行被替换时才写主存。

✅ 关键理解：Write-Back 的写流量与写操作次数无关，而与替换频率和脏行比例有关。
