虚拟内存

知识点 1 · 地址空间（Address Space）的抽象

🎯 为什么研究这个问题

在早期计算机中，程序直接使用物理地址访问内存。这带来了严重问题：多个程序同时运行时会互相覆盖内存；程序必须在编译时知道自己被加载到哪个地址；一个错误的程序可以破坏整个系统。

虚拟内存通过引入一层地址空间抽象，彻底解决了这些问题。每个进程拥有自己独立的虚拟地址空间，认为自己独占整个内存——从地址 0 到地址上限。操作系统和硬件（MMU）负责将虚拟地址翻译为物理地址，进程完全不知道自己实际使用了哪些物理内存。

虚拟内存是现代操作系统最关键的基础设施之一，没有它就没有进程隔离、内存保护和按需分页。

🔍 抽象化方法

虚拟内存系统涉及三层地址概念：

① 虚拟地址（Virtual Address, VA）：程序使用的地址，由 CPU 生成。每个进程有独立的虚拟地址空间，32 位系统为 \(2^{32} = 4\text{GB}\)，64 位系统理论上为 \(2^{64}\)（实际通常使用 48 位，即 256TB）。

② 物理地址（Physical Address, PA）：实际的内存芯片地址，由 MMU 翻译后输出到地址总线。物理内存通常远小于虚拟地址空间。

③ 磁盘地址：当物理内存不足时，部分页面被交换（swap）到磁盘。虚拟内存使得程序可以使用比物理内存更大的地址空间。

虚拟内存提供三大功能：地址翻译（VA → PA）、内存保护（不同进程不能访问彼此的内存）、按需分页（只将实际使用的页面加载到物理内存）。

📐 公式推导 / 模型建立

分页系统将虚拟和物理地址空间都划分为固定大小的页面（Page），典型大小为 4KB。

地址分解：

\[ \text{VA} = \underbrace{\text{VPN}}_{\text{虚拟页号}} \| \underbrace{\text{Offset}}_{\text{页内偏移}} \]

对于 32 位地址、4KB 页面：偏移量 = \(\log_2(4096) = 12\) 位，VPN = \(32 - 12 = 20\) 位。

地址翻译的核心操作：

\[ \text{PA} = \text{PageTable}[\text{VPN}].\text{PPN} \| \text{Offset} \]

即用虚拟页号查页表得到物理页号（PPN），与页内偏移拼接形成物理地址。偏移量在翻译过程中保持不变。

虚拟地址空间大小与页表项数量的关系：

\[ \text{页表项数} = \frac{2^{\text{VA位数}}}{2^{\text{偏移位数}}} = 2^{\text{VPN位数}} \]
⚠️ 虚拟地址空间大小 ≠ 物理内存大小！虚拟地址空间由地址位数决定（如 32 位 = 4GB），而物理内存可以更小（如 2GB）甚至更大。虚拟内存正是通过按需分页和磁盘交换来弥补这种差异。
🌐 实际应用

进程隔离：每个进程有独立的页表，因此进程 A 的虚拟地址 0x1000 和进程 B 的 0x1000 映射到不同的物理页面。一个进程崩溃不会影响其他进程。

共享库：多个进程可以将同一个物理页面映射到各自的虚拟地址空间，实现代码共享（如 libc.so），节省物理内存。

内存映射文件（mmap）：将文件直接映射到虚拟地址空间，通过读写内存来操作文件，由操作系统自动处理文件 I/O 和缓存。

写时复制（Copy-on-Write）：fork() 时父子进程共享同一物理页面，标记为只读。只有当某一方写入时才触发页面复制，大幅减少 fork 开销。

✏️ 例题分析与易错点

题目：32 位虚拟地址空间，页大小 4KB，物理内存 256MB。求：① VPN 位数；② 偏移量位数；③ PPN 位数；④ 物理地址位数。

解答：

① 偏移量 = \(\log_2(4\text{KB}) = \log_2(2^{12}) = 12\) 位

② VPN = \(32 - 12 = 20\) 位

③ 物理地址位数 = \(\log_2(256\text{MB}) = \log_2(2^{28}) = 28\) 位

④ PPN = \(28 - 12 = 16\) 位

❌ 常见错误：混淆 VPN 位数和 PPN 位数。VPN 由虚拟地址位数决定，PPN 由物理地址位数决定，两者通常不同！

✅ 正确做法：始终分别根据虚拟/物理地址空间大小来计算各自的页号位数。

知识点 2 · 页表（Page Table）与地址翻译

🎯 为什么研究这个问题

地址空间抽象告诉我们"虚拟地址需要翻译为物理地址"，但具体怎么翻译？答案是页表（Page Table）——一个由操作系统维护、由硬件查询的映射表。

页表是虚拟内存系统的核心数据结构。它不仅存储地址映射，还包含访问权限（读/写/执行）、脏位（Dirty Bit）、引用位（Reference Bit）等元数据，是内存保护和页面置换的基础。

理解页表的结构和查询过程，是理解操作系统内存管理的关键。

🔍 抽象化方法

页表是一个以虚拟页号（VPN）为索引的数组，每个条目称为页表项（Page Table Entry, PTE）。

PTE 的典型字段：

① 有效位（Valid Bit）：该虚拟页是否有合法映射。为 0 则访问触发缺页异常。

② 物理页号（PPN）：映射到的物理页帧号。

③ 权限位：R（可读）、W（可写）、X（可执行）。违反权限触发保护异常。

④ 脏位（Dirty Bit）：该页是否被写过。用于页面置换时决定是否需要写回磁盘。

⑤ 引用位（Reference/Access Bit）：该页是否被访问过。用于页面置换算法（如 Clock 算法）。

每个进程有自己的页表。在 RISC-V 中，satp 寄存器存储当前进程页表的物理基地址。进程切换时，操作系统更新 satp 并刷新 TLB。

📐 公式推导 / 模型建立

地址翻译的完整过程：

\[ \text{PTE} = \text{PageTable}[\text{VPN}] = \text{Mem}[\text{PTBR} + \text{VPN} \times \text{sizeof(PTE)}] \]

其中 PTBR（Page Table Base Register）是页表基地址寄存器（RISC-V 中的 satp）。

如果 PTE.Valid = 1：

\[ \text{PA} = \text{PTE.PPN} \| \text{Offset} \]

如果 PTE.Valid = 0，触发缺页异常（Page Fault）。

页表大小计算：

\[ \text{页表大小} = 2^{\text{VPN位数}} \times \text{sizeof(PTE)} \]

例：32 位地址、4KB 页面、4 字节 PTE → 页表大小 = \(2^{20} \times 4 = 4\text{MB}\)。每个进程 4MB，100 个进程就是 400MB——仅仅存储页表就占用大量内存！这就是需要多级页表的原因。

⚠️ 页表本身存储在物理内存中！每次地址翻译都需要先访问内存读取 PTE，这意味着每次内存访问变成了两次：一次查页表，一次访问数据。没有 TLB 的话，性能会下降一半。
🌐 实际应用

内存保护：通过 PTE 的权限位，操作系统可以将代码段标记为只读+可执行（R+X），数据段标记为可读写（R+W），栈标记为可读写但不可执行（防止代码注入攻击，即 NX/DEP 保护）。

按需分页（Demand Paging）：程序启动时并不将所有页面加载到内存，而是将页表项标记为无效。首次访问时触发缺页异常，操作系统再加载对应页面。这使得程序启动更快、内存使用更高效。

页面置换：当物理内存不足时，操作系统选择一个页面淘汰（写回磁盘如果是脏页），释放页帧给新页面。常用算法有 LRU（Least Recently Used）、Clock、FIFO 等。

✏️ 例题分析与易错点

题目：虚拟地址 0x00003A7C，页大小 4KB。页表内容如下：VPN 3 → PPN 0x0F，Valid=1。求物理地址。

解答：

① 偏移量 = 低 12 位 = 0xA7C

② VPN = 0x00003A7C >> 12 = 0x3

③ 查页表：VPN 3 → PPN = 0x0F

④ 物理地址 = PPN | Offset = 0x0FA7C

❌ 常见错误：忘记偏移量在翻译中保持不变，错误地将整个虚拟地址替换为物理地址。

✅ 正确做法：只替换高位的页号部分，低位偏移量直接保留。用位拼接而非加法：PA = PPN << 12 | Offset。

知识点 3 · TLB（Translation Lookaside Buffer）

🎯 为什么研究这个问题

上一节我们看到，每次内存访问都需要先查页表（在内存中），这意味着一次数据访问变成了至少两次内存访问——性能直接减半。这在高性能处理器中是不可接受的。

TLB（Translation Lookaside Buffer，翻译后备缓冲器）是解决这一问题的关键硬件：它是一个小型、高速的缓存，专门存储最近使用的页表项。TLB 命中时，地址翻译在一个时钟周期内完成，无需访问内存中的页表。

由于程序具有很强的地址局部性，TLB 的命中率通常高达 99% 以上，因此绝大多数内存访问不会受到页表查询的性能惩罚。

🔍 抽象化方法

TLB 的结构类似于缓存（Cache），但存储的不是数据而是地址翻译映射（VPN → PPN + 权限位）。

TLB 查找过程：

① CPU 生成虚拟地址，提取 VPN

② 用 VPN 在 TLB 中并行查找（全相联或组相联）

③ TLB Hit：找到匹配的 VPN，直接输出 PPN → 快速完成翻译

④ TLB Miss：未找到 → 执行页表遍历（Page Table Walk），将结果写入 TLB，然后重试

TLB 通常很小（32-128 条目），但由于全相联结构和超高命中率，对性能的提升是决定性的。典型的 TLB 访问延迟约 0.5-1 个时钟周期，而内存访问可能需要 100+ 个周期。

📐 公式推导 / 模型建立

TLB 对有效内存访问时间（EMAT）的影响：

\[ \text{EMAT} = h \times (t_{\text{TLB}} + t_{\text{mem}}) + (1 - h) \times (t_{\text{TLB}} + t_{\text{PT}} + t_{\text{mem}}) \]

其中 \(h\) 是 TLB 命中率，\(t_{\text{TLB}}\) 是 TLB 查找时间，\(t_{\text{PT}}\) 是页表遍历时间，\(t_{\text{mem}}\) 是实际数据的内存访问时间。

简化：

\[ \text{EMAT} = t_{\text{TLB}} + t_{\text{mem}} + (1 - h) \times t_{\text{PT}} \]

例：\(t_{\text{TLB}} = 1\text{ns}\)，\(t_{\text{mem}} = 100\text{ns}\)，\(t_{\text{PT}} = 100\text{ns}\)（一次额外内存访问），\(h = 99\%\)：

\[ \text{EMAT} = 1 + 100 + 0.01 \times 100 = 102\text{ns} \]

对比无 TLB：\(100 + 100 = 200\text{ns}\)。TLB 将有效访问时间从 200ns 降到 102ns，性能几乎翻倍！

TLB 覆盖范围：

\[ \text{TLB Reach} = \text{TLB 条目数} \times \text{页面大小} \]

例：64 条目 × 4KB = 256KB。如果程序的工作集超过 TLB Reach，命中率会显著下降。

⚠️ 进程切换时必须处理 TLB 一致性！因为不同进程的 VPN 可能映射到不同的 PPN。方案一：切换时刷新（flush）整个 TLB——简单但代价高。方案二：为 TLB 条目加上 ASID（Address Space ID）标签，允许多个进程的映射共存。RISC-V 的 satp 寄存器包含 ASID 字段。
🌐 实际应用

分级 TLB：现代处理器通常有两级 TLB：L1 iTLB（指令，32-64 条目，全相联）+ L1 dTLB（数据，32-64 条目）+ L2 TLB（统一，512-4096 条目，组相联）。

大页（Huge Pages）：使用 2MB 或 1GB 的大页可以显著增大 TLB Reach。例如 64 条目 × 2MB = 128MB，足以覆盖许多数据库的工作集。Linux 中使用 HugePages 或 Transparent Huge Pages（THP）。

TLB Shootdown：在多核系统中，当操作系统修改页表时，必须通知所有其他核心使其 TLB 中对应条目失效（通过 IPI，处理器间中断）。这是多核系统的重要性能瓶颈之一。

✏️ 例题分析与易错点

题目：TLB 访问 1ns，页表遍历 80ns，内存访问 100ns。TLB 命中率 95%。求 EMAT。如果使用大页将命中率提升到 99.5%，EMAT 变为多少？

解答：

原始：\(\text{EMAT} = 1 + 100 + (1-0.95) \times 80 = 101 + 4 = 105\text{ns}\)

大页后：\(\text{EMAT} = 1 + 100 + (1-0.995) \times 80 = 101 + 0.4 = 101.4\text{ns}\)

性能提升：\((105 - 101.4) / 105 \approx 3.4\%\)

❌ 常见错误：忽略 TLB 自身的访问时间（即使命中也要 1ns），或者错误地认为 TLB Miss 时不需要 TLB 查找时间。

✅ 正确做法：TLB 查找时间在命中和未命中的情况下都会产生（串行/并行取决于架构），按公式正确代入。

知识点 4 · 多级页表（Multi-level Page Table）

🎯 为什么研究这个问题

单级页表的致命问题是空间浪费。以 32 位地址、4KB 页面为例，单级页表需要 \(2^{20}\) 个页表项，每个 4 字节，共 4MB。但实际程序可能只使用了很小一部分虚拟地址空间（如代码段、数据段、堆、栈之间有大量空洞）。为空洞区域分配页表项完全是浪费。

对于 64 位系统，问题更加严峻：\(2^{48}\) 字节的虚拟地址空间（48 位有效地址），4KB 页面，需要 \(2^{36}\) 个页表项——这需要 256GB 仅用来存储页表，显然不可接受。

多级页表通过分层索引解决了这一问题：只为实际使用的地址范围分配页表空间。

🔍 抽象化方法

多级页表的核心思想是将 VPN 分成多段，每一段作为一级页表的索引。

以二级页表为例（32 位地址、4KB 页面）：

VPN（20 位）被分为：一级索引 VPN1（10 位）+ 二级索引 VPN2（10 位）

翻译过程：

① 用 VPN1 索引一级页表（Page Directory），得到二级页表的物理基地址

② 用 VPN2 索引二级页表，得到最终的 PPN

③ PPN 与 Offset 拼接得到物理地址

空间节省的关键：如果一级页表中某个条目的 Valid=0，意味着整个对应的二级页表不需要存在。对于大量未使用的虚拟地址范围，只需要在一级页表中记录"无效"即可，无需分配二级页表。

RISC-V Sv39（用于 64 位系统）使用三级页表，Sv48 使用四级页表。x86-64 使用四级页表（PML4 → PDPT → PD → PT），最新的 5-level paging 使用五级。

📐 公式推导 / 模型建立

二级页表地址翻译的形式化描述：

\[ \text{PDE} = \text{Mem}[\text{PTBR} + \text{VPN1} \times \text{sizeof(PTE)}] \] \[ \text{PTE} = \text{Mem}[\text{PDE.PPN} \times \text{PageSize} + \text{VPN2} \times \text{sizeof(PTE)}] \] \[ \text{PA} = \text{PTE.PPN} \| \text{Offset} \]

空间分析（32 位、4KB 页面、4 字节 PTE）：

单级页表：\(2^{20} \times 4 = 4\text{MB}\)（固定开销）

二级页表：一级页表 = \(2^{10} \times 4 = 4\text{KB}\)（始终需要）+ 每个使用的二级页表 = \(2^{10} \times 4 = 4\text{KB}\)

如果程序只使用了 \(k\) 个二级页表范围：

\[ \text{二级页表总大小} = 4\text{KB} + k \times 4\text{KB} \]

例如程序只使用了 4 个区域（代码、数据、堆、栈）：\(4 + 4 \times 4 = 20\text{KB}\)，对比单级的 4MB，节省了 \(99.5\%\)！

时间代价：k 级页表需要 k 次内存访问完成翻译。

\[ T_{\text{translate}} = k \times t_{\text{mem}} \quad (\text{TLB Miss 时}) \]
⚠️ 多级页表是空间换时间的反向操作——用更多的内存访问次数换取更少的页表内存占用。TLB 的存在使得这一代价在实际中可以接受：99%+ 的翻译由 TLB 直接完成，不需要多次内存访问。没有 TLB 的多级页表将会非常慢！
🌐 实际应用

RISC-V Sv39：使用三级页表，虚拟地址 39 位（512GB 地址空间）。VPN 分为三段各 9 位，加上 12 位偏移。每级页表刚好一页（512 个 8 字节 PTE = 4KB）。这种设计使页表页面本身可以用一个物理页帧存储。

x86-64 四级页表：PML4（Page Map Level 4）→ PDPT → PD → PT，每级 512 个条目（9 位索引），共支持 48 位虚拟地址（256TB）。CR3 寄存器存储 PML4 的物理地址。

内核页表映射：在 Linux 中，内核拥有所有进程共享的页表映射（内核空间），通常位于虚拟地址的高半部分。进程切换时只需切换用户空间的页表映射，内核部分保持不变。

超大页与页表层级：多级页表天然支持超大页——只需在中间层级的页表项中直接映射大页（如 2MB 或 1GB），无需展开到下一级。这由 PTE 中的特殊位（如 Leaf 位）标记。

✏️ 例题分析与易错点

题目：RISC-V Sv39 系统，虚拟地址 0x0000_003F_F000_1A80。页大小 4KB。求：① 各级 VPN 的值；② 需要几次内存访问完成翻译（TLB Miss）。

解答：

Sv39 地址格式：VPN[2](9位) | VPN[1](9位) | VPN[0](9位) | Offset(12位)

① 0x3FF0001A80 的二进制低 39 位：

Offset = 0xA80 (低12位)

VPN[0] = 0x001 (接下来9位)

VPN[1] = 0x100 (再9位) = 0x000

VPN[2] = 0x0FF (最高9位)

② TLB Miss 时需要 3 次内存访问（三级页表每级一次）+ 1 次访问实际数据 = 共 4 次内存访问。

❌ 常见错误：忘记计算最终数据访问的内存时间，只算了页表遍历的 3 次。实际 EMAT 计算必须加上数据本身的访问时间。

✅ 正确做法：总内存访问 = 页表级数 + 1（数据访问），全部由 TLB 是否命中来决定是否需要页表遍历。
