Chapitre 3

中断与 I/O

知识点 1 · 轮询（Polling）vs 中断（Interrupt）vs DMA

🎯 为什么研究这个问题

CPU 需要与各种 I/O 设备（键盘、磁盘、网卡、显示器……）进行通信。但 CPU 运行速度极快（GHz 级别），而大多数 I/O 设备的速度要慢几个数量级——一次磁盘访问耗时约 10ms，相当于 CPU 执行数千万条指令的时间。如何高效地协调这种巨大的速度差异？

计算机体系结构提供了三种根本不同的 I/O 交互方式：轮询（Polling）、中断（Interrupt）和 DMA（Direct Memory Access）。它们在 CPU 利用率、延迟、复杂度方面有截然不同的特征。理解这三种方式的优劣和适用场景，是系统编程和操作系统设计的基础。

更重要的是，现代高性能系统往往混合使用这三种方式——例如 Linux 网络子系统的 NAPI 机制就在中断和轮询之间动态切换，以应对从低负载到线速的不同流量场景。只有深刻理解每种方式的本质，才能做出正确的设计权衡。

🔍 抽象化方法

三种 I/O 交互方式的核心思想可以用日常类比理解：

① 轮询（Polling）——"不停看手机"：CPU 反复检查设备的状态寄存器（Status Register），看数据是否准备好。如果没有，继续检查；如果有，读取数据。这就像你每隔几秒看一次手机，等待消息到来。实现简单，但在等待期间 CPU 完全被占用（busy waiting），无法做其他工作。

② 中断（Interrupt）——"开通知推送"：CPU 告诉设备"数据好了就通知我"，然后去做其他任务。当设备准备好数据后，通过中断信号通知 CPU，CPU 暂停当前任务、处理 I/O 数据、再恢复原任务。就像手机推送通知——你专心做事，消息来了手机震动提醒你。效率高，但中断处理有上下文切换的开销。

③ DMA（Direct Memory Access）——"快递直送"：对于大块数据传输，让专门的 DMA 控制器在设备和内存之间直接搬运数据，完全不经过 CPU。CPU 只需告诉 DMA 控制器"从哪搬、搬到哪、搬多少"，然后就可以去做其他工作。传输完成后 DMA 用中断通知 CPU。就像网购——你下单（CPU 编程 DMA），快递公司直接把货送到家（DMA 传输到内存），送达后通知你（中断）。

核心对比：轮询 = 忙等待（busy waiting），中断 = 事件驱动（event-driven），DMA = 硬件卸载（hardware offloading）。

📐 公式推导 / 模型建立

我们可以定量分析三种方式对 CPU 利用率的影响。假设设备每 \(T_d\) 个时钟周期产生一次数据：

轮询方式：每次轮询检查花费 \(t_p\) 个周期。如果 CPU 持续轮询，则：

\[ \text{CPU 利用率}_{\text{poll}} = 1 - \frac{t_p}{T_d} \]

当设备很慢（\(T_d\) 很大）时，大部分时间在无效检查上浪费——利用率接近 0。

中断方式：每次中断处理花费 \(t_{\text{int}}\) 个周期（包括上下文保存/恢复）。中断只在数据就绪时触发：

\[ \text{CPU 利用率}_{\text{int}} = 1 - \frac{t_{\text{int}}}{T_d} \]

当 \(T_d\) 较大时（慢速设备），\(t_{\text{int}} / T_d\) 很小，CPU 利用率很高——中断远优于轮询。但当 \(T_d\) 很小时（高速设备），频繁的中断开销 \(t_{\text{int}}\) 可能大于轮询开销 \(t_p\)，此时轮询反而更好。

DMA 方式：CPU 只需花费 DMA 设置时间 \(t_{\text{setup}}\) 和传输完成后的中断处理时间 \(t_{\text{int}}\)。数据传输时间由总线带宽决定：

\[ T_{\text{transfer}} = \frac{n \times \text{word\_size}}{B_{\text{bus}}} \]

CPU 在整个 \(T_{\text{transfer}}\) 期间都是空闲的，可以执行其他任务。这使 DMA 成为大块数据传输的最佳选择。

⚠️ 中断并不总是比轮询好！对于极高频事件（例如万兆网卡以 10 Gbps 速率每秒接收数百万个数据包），每个包都触发中断会导致中断处理开销压垮 CPU，产生中断活锁（Interrupt Livelock）——CPU 把所有时间都花在处理中断上，无法执行实际的网络协议栈代码。现代系统使用混合方法：Linux 的 NAPI 机制在低流量时使用中断，高流量时切换为轮询，实现最优平衡。
🌐 实际应用

网络数据包处理（NAPI）：Linux 内核的 NAPI（New API）是中断与轮询的混合方案。收到第一个包时触发中断，然后关闭中断、切换到轮询模式批量处理后续数据包。当队列清空后重新启用中断。这种方式在低延迟和高吞吐之间取得了优秀的平衡。

键盘输入（中断驱动）：键盘是典型的低速设备——用户每秒最多按几十次键。用中断方式最合适：CPU 平时执行其他任务，按键时硬件触发中断，CPU 读取键码后立即恢复原任务。用轮询方式完全是浪费。

磁盘 I/O（DMA）：磁盘每次读取以扇区（512 字节或 4KB）为单位，让 CPU 逐字搬运极其浪费。DMA 控制器直接将数据从磁盘控制器传输到内存缓冲区，传输完成后中断通知 CPU。现代 NVMe SSD 也使用 DMA 进行高带宽数据传输。

实时系统（轮询优先）：在对延迟有严格要求的实时系统中（如飞控系统），轮询的确定性响应时间可能比中断的不确定延迟更可取。中断的响应时间取决于当前指令完成时间和中断优先级，而轮询的延迟是可预测的。

✏️ 例题分析与易错点

题目：一个磁盘控制器每 10ms 产生 1 个数据块。轮询检查一次花费 100ns，中断处理一次花费 5μs。分别计算两种方式下 CPU 的利用率，并判断哪种更优。

解答：

首先统一单位。\(T_d = 10\,\text{ms} = 10{,}000{,}000\,\text{ns}\)，\(t_p = 100\,\text{ns}\)，\(t_{\text{int}} = 5{,}000\,\text{ns}\)。

但注意：轮询方式下 CPU 必须在整个 \(T_d\) 期间持续检查（busy waiting），不仅仅是一次检查。CPU 一直在轮询循环中，因此在等待期间 CPU 利用率（做有用工作）接近 0%。严格来说：

\[ \text{CPU 利用率}_{\text{poll}} \approx 0\%\;(\text{忙等待期间完全浪费}) \]

中断方式：CPU 只在中断发生时花费 5μs 处理，其余时间自由：

\[ \text{CPU 利用率}_{\text{int}} = 1 - \frac{5{,}000}{10{,}000{,}000} = 1 - 0.0005 = 99.95\% \]

结论：对于这种慢速设备，中断方式远优于轮询——CPU 利用率从接近 0% 提升到 99.95%。

❌ 常见错误：混淆"轮询一次的时间"和"轮询等待期间的总开销"。轮询的问题不是单次检查费时，而是 CPU 被锁定在轮询循环中无法做其他工作。

✅ 正确理解：评估 I/O 方式时，关键指标是 CPU 可以用于其他任务的时间比例，而非单次操作的开销。

知识点 2 · 中断处理流程与向量表

🎯 为什么研究这个问题

当中断发生时，CPU 必须完成一项看似矛盾的任务：在不丢失当前工作的前提下，立即去处理另一件事，然后无缝恢复原来的工作。这就好比你正在写论文，突然电话响了——你需要记住写到哪里（保存现场）、接电话（处理中断）、挂断后回到之前的位置继续写（恢复现场）。

这个过程的每一步都必须精确无误：如果保存现场不完整，恢复后程序会产生不可预测的错误；如果中断响应太慢，可能丢失数据或错过实时截止时间；如果中断处理程序本身有 Bug，整个系统可能崩溃。

中断机制是操作系统的基石——定时器中断驱动进程调度，I/O 中断驱动设备管理，异常/陷阱（Trap）实现系统调用和内存保护。没有中断，就没有现代操作系统。

🔍 抽象化方法

中断处理的完整流程可以分为以下步骤：

① 设备触发中断：I/O 设备（如键盘、网卡、定时器）通过中断线（Interrupt Line）向 CPU 发送中断请求信号。

② CPU 完成当前指令：CPU 不会在指令执行到一半时响应中断——它会先完成当前正在执行的指令。这保证了 CPU 状态的一致性。

③ 硬件自动保存关键状态：CPU 硬件自动保存程序计数器（PC）和状态寄存器。在 RISC-V 中：PC 保存到 mepc 寄存器，中断原因保存到 mcause，全局中断被关闭（mstatus.MIE 清零，防止嵌套中断）。

④ 查中断向量表：CPU 读取中断向量号，通过中断向量表（Interrupt Vector Table）找到对应的中断服务程序（ISR，Interrupt Service Routine）的地址。向量表本质上是一个 ISR 地址数组，索引就是中断号。RISC-V 中 mtvec 寄存器存储向量表基地址。

⑤ 跳转到 ISR 执行：CPU 跳转到 ISR 地址开始执行。ISR 首先保存其他需要使用的寄存器（软件保存），然后执行具体的中断处理逻辑。

⑥ ISR 恢复上下文并返回：ISR 恢复保存的寄存器，执行 mret（Machine Return）指令返回被中断的程序。mret 同时恢复 mepc 到 PC、恢复中断使能状态。

中断类型：外部中断（External Interrupt，来自 I/O 设备）、软件中断（Software Interrupt，由 ecall 指令触发，用于系统调用）、异常（Exception/Trap，如非法指令、页缺失）。外部中断是异步的（随时可能发生），异常是同步的（由当前指令触发）。

📐 公式推导 / 模型建立

中断延迟（Interrupt Latency）是从中断信号到达到 ISR 第一条指令开始执行的时间：

\[ T_{\text{latency}} = T_{\text{finish\_instr}} + T_{\text{save\_context}} + T_{\text{vector\_lookup}} + T_{\text{jump}} \]

其中 \(T_{\text{finish\_instr}}\) 是完成当前指令的时间（最坏情况为最长指令的执行时间），\(T_{\text{save\_context}}\) 是硬件保存 PC 和状态的时间，\(T_{\text{vector\_lookup}}\) 是查向量表的时间，\(T_{\text{jump}}\) 是跳转到 ISR 的时间。

RISC-V 向量模式地址计算：当 mtvec 的 MODE 字段为 1（向量模式）时，不同中断跳转到不同的地址：

\[ \text{ISR 地址} = \text{mtvec.BASE} + 4 \times \text{cause} \]

例如：若 mtvec.BASE = 0x80000000，定时器中断（cause = 7）的 ISR 地址为：

\[ \text{0x80000000} + 4 \times 7 = \text{0x8000001C} \]

上下文保存/恢复开销：若 ISR 需要保存 \(N\) 个寄存器，每次内存访问耗时 \(t_{\text{mem}}\)，则上下文保存和恢复的总开销为：

\[ T_{\text{context}} = 2N \times t_{\text{mem}} \quad (\text{save} + \text{restore}) \]

这就是为什么 ISR 应该尽量少用寄存器——保存的寄存器越少，中断处理越快。

⚠️ 中断处理程序必须尽可能短！在非嵌套中断系统中，ISR 执行期间全局中断是关闭的，长时间运行的 ISR 会阻塞其他中断，导致数据丢失或实时系统错过截止时间。正确的做法是：ISR 只做最少量的紧急工作（确认设备、将数据放入缓冲区），然后调度延迟处理（如 Linux 的 Bottom Half / Tasklet / Workqueue）来完成耗时操作。
🌐 实际应用

定时器中断与进程调度：操作系统通过定时器中断（通常以 100–1000 Hz 频率触发）来实现进程的时间片轮转调度。每次定时器中断到来，OS 检查当前进程是否用完了时间片，如果是，就切换到下一个进程。这就是为什么即使只有一个 CPU 核心，多个程序也能"同时"运行——定时器中断驱动的快速切换制造了并发的假象。

键盘/鼠标中断处理：当用户按键时，键盘控制器触发中断。ISR 读取键码（scan code），放入内核缓冲区，然后唤醒等待输入的进程。整个 ISR 非常短——只需要几微秒。

页缺失（Page Fault）异常：当 CPU 访问一个尚未映射到物理内存的虚拟地址时，MMU 触发页缺失异常。OS 的异常处理程序从磁盘加载所需的页面到物理内存，更新页表，然后重新执行引起缺失的指令。这是虚拟内存实现的核心机制。

系统调用（ecall）：在 RISC-V 中，用户程序通过 ecall 指令触发一个同步异常（Environment Call），从用户态陷入内核态。内核根据 a7 寄存器中的系统调用号执行相应的内核函数（如 read、write、fork），完成后通过 sret 返回用户态。

✏️ 例题分析与易错点

题目：假设 mtvec = 0x80000000（向量模式），定时器中断的 cause 编号为 7。(1) 计算定时器 ISR 的入口地址。(2) 写出 RISC-V 中断入口和退出的汇编代码框架。

解答（1）：

\[ \text{ISR 地址} = \text{0x80000000} + 4 \times 7 = \text{0x8000001C} \]

解答（2）中断入口/退出框架：

# ─── 中断入口（ISR prologue）───
timer_isr:
    addi sp, sp, -32      # 分配栈空间
    sw   t0, 0(sp)        # 保存 ISR 中要使用的寄存器
    sw   t1, 4(sp)
    sw   t2, 8(sp)
    sw   a0, 12(sp)

    # ─── 中断处理主体 ───
    # ... 读取设备数据、更新计数器等 ...

    # ─── 中断退出（ISR epilogue）───
    lw   t0, 0(sp)        # 恢复寄存器
    lw   t1, 4(sp)
    lw   t2, 8(sp)
    lw   a0, 12(sp)
    addi sp, sp, 32       # 释放栈空间
    mret                   # 返回被中断程序
                           # (硬件自动从 mepc 恢复 PC)

❌ 常见错误：忘记在 ISR 中保存临时寄存器！虽然 t0–t6 在函数调用中是 caller-saved，但在中断处理中所有使用的寄存器都必须保存——因为被中断的程序不知道中断的存在，没有机会保存自己的寄存器。

✅ 关键区别：函数调用有调用约定（caller/callee-saved），但中断对被中断的程序完全透明。ISR 必须保存并恢复它使用的每一个寄存器，不管它是 t 寄存器还是 s 寄存器。

知识点 3 · DMA（Direct Memory Access）原理

🎯 为什么研究这个问题

考虑一次典型的磁盘读取：将 4KB 的数据块从磁盘控制器搬运到内存。如果由 CPU 逐字（4 字节）搬运，CPU 需要执行 1024 次 load + 1024 次 store = 2048 条指令。在此期间，CPU 被完全占用——做的却是最简单的"搬运工"工作，这些宝贵的计算能力本可以用于执行用户程序。

DMA（Direct Memory Access，直接内存访问）的思想很简单但很强大：让专门的硬件控制器代替 CPU 来搬运数据。CPU 只需要告诉 DMA 控制器"从哪搬、搬到哪、搬多少"，然后就可以去做其他工作。DMA 控制器独立完成整个数据传输过程，完成后用中断通知 CPU。

在现代系统中，几乎所有大块数据传输都使用 DMA：磁盘读写、网卡收发数据包、GPU 显存访问、声卡音频流、USB 传输等。理解 DMA 对于编写高性能 I/O 代码和理解操作系统 I/O 子系统至关重要。

🔍 抽象化方法

DMA 控制器是一个独立的硬件模块，包含以下关键寄存器：

① 源地址寄存器（Source Address Register）：数据从哪里读取（可以是设备端口地址或内存地址）。

② 目标地址寄存器（Destination Address Register）：数据写到哪里（通常是内存地址）。

③ 字节计数寄存器（Byte Count Register）：需要传输的总字节数。每传输一个字，计数器减一。减到零时传输完成。

④ 控制/状态寄存器（Control/Status Register）：配置传输方向、启动/停止传输、指示传输状态和完成情况。

DMA 传输过程：

Step 1：CPU 向 DMA 控制器的寄存器写入源地址、目标地址和字节数（编程 DMA）。

Step 2：CPU 启动 DMA 传输（写入控制寄存器），然后 CPU 自由执行其他任务。

Step 3：DMA 控制器取得总线控制权，逐字在设备和内存之间传输数据。

Step 4：传输完成后（计数器归零），DMA 控制器向 CPU 发送中断信号通知传输完成。

总线仲裁模式：DMA 和 CPU 共享系统总线，不能同时使用。周期窃取（Cycle Stealing）：DMA 每次只借用总线传输一个字，然后归还，对 CPU 影响小但传输慢。突发模式（Burst Mode）：DMA 占用总线完成整个传输块，速度快但会暂时阻塞 CPU 的总线访问。块模式（Block Mode）：与突发类似，但在块边界处释放总线。

📐 公式推导 / 模型建立

CPU 搬运方式（无 DMA）的耗时：CPU 用循环逐字复制 \(N\) 个字，每个字需要一次 load + 一次 store + 循环控制指令：

\[ T_{\text{CPU\_copy}} = N \times (t_{\text{load}} + t_{\text{store}} + t_{\text{loop}}) \]

在此期间 CPU 被完全占用，无法执行其他任务。

DMA 方式的耗时：CPU 只需花时间设置 DMA（编程寄存器）和处理完成中断：

\[ T_{\text{DMA}} = t_{\text{setup}} + \frac{N \times \text{word\_size}}{B_{\text{bus}}} + t_{\text{interrupt}} \]

其中 \(t_{\text{setup}}\) 是 DMA 设置时间（通常几条指令），\(B_{\text{bus}}\) 是总线带宽（bytes/s），\(t_{\text{interrupt}}\) 是完成中断处理时间。关键是：在 \(\frac{N \times \text{word\_size}}{B_{\text{bus}}}\) 这段传输时间内，CPU 是完全空闲的，可以执行其他任务！

CPU 时间节省（CPU 从搬运中释放的时间）：

\[ \text{Speedup}_{\text{CPU}} \approx \frac{T_{\text{CPU\_copy}}}{t_{\text{setup}} + t_{\text{interrupt}}} \]

对于大块传输（\(N\) 很大），这个加速比可以达到几百倍甚至几千倍——CPU 从"搬运工"变为"管理者"。

⚠️ DMA 引入缓存一致性问题！DMA 控制器直接读写主内存，绕过 CPU 缓存。这导致两个问题：① DMA 写入内存后，CPU 缓存中可能还是旧数据（stale cache）——CPU 读到过期值；② CPU 修改了缓存中的数据但还没写回内存，DMA 从内存读到的是旧值。解决方案：DMA 读取前执行 cache flush（将脏数据写回内存）；DMA 写入后执行 cache invalidate（丢弃缓存中的旧数据）；或者使用硬件一致性 DMA（通过缓存嗅探协议自动维护一致性）。
🌐 实际应用

磁盘控制器 DMA：现代 SATA/NVMe 存储控制器全部使用 DMA。OS 将读写请求以描述符（descriptor）形式放入内存中的命令队列，设备控制器通过 DMA 读取命令、执行 I/O、通过 DMA 将数据直接写入 OS 指定的内存缓冲区，最后发送 MSI 中断通知完成。整个过程 CPU 只需要填写描述符和处理完成中断。

网卡（NIC）DMA：网卡使用环形缓冲区（Ring Buffer）+ DMA 实现零拷贝收发。OS 预先分配 DMA 缓冲区并将地址写入网卡描述符环。当数据包到达时，网卡 DMA 直接将数据写入对应的缓冲区，更新描述符状态，然后发中断通知 CPU。这种设计支持 10–100 Gbps 的线速处理。

GPU 显存传输（PCIe DMA）：GPU 通过 PCIe 总线的 DMA 在系统内存和显存（VRAM）之间高速传输纹理、顶点数据和计算结果。CUDA 的 cudaMemcpy 底层就是 PCIe DMA 操作。

Scatter-Gather DMA：高级 DMA 控制器支持 scatter-gather 操作——一次 DMA 传输可以从多个不连续的内存区域读取（gather）或写入多个不连续的区域（scatter）。这对虚拟内存系统特别重要，因为一个连续的虚拟缓冲区在物理内存中可能分布在不同的页面上。

✏️ 例题分析与易错点

题目：需要传输 4096 字节数据，总线带宽为 1 GB/s。CPU 逐字（4 字节）复制每个字需要 5ns（load + store + loop）。DMA 设置时间为 1μs，完成中断处理时间为 2μs。比较两种方式的时间和 CPU 占用情况。

解答：

数据量：4096 字节 = 1024 个字（4 字节/字）

CPU 逐字复制：

\[ T_{\text{CPU\_copy}} = 1024 \times 5\,\text{ns} = 5120\,\text{ns} = 5.12\,\mu\text{s} \]

CPU 占用时间 = 5.12μs（100% 占用）

DMA 方式：

\[ T_{\text{transfer}} = \frac{4096}{1 \times 10^9} = 4.096\,\mu\text{s} \] \[ T_{\text{DMA\_total}} = 1\,\mu\text{s} + 4.096\,\mu\text{s} + 2\,\mu\text{s} = 7.096\,\mu\text{s} \]

CPU 占用时间 = 1μs（setup）+ 2μs（interrupt）= 3μs

CPU 释放时间 = 5.12 - 3 = 2.12μs。虽然总时间 DMA 反而略多，但 CPU 在 4.096μs 的传输期间可以做其他工作！对于更大的数据传输，优势更加明显。

❌ 常见错误：只比较总完成时间，忽略 CPU 被释放的时间。DMA 的价值不在于"更快完成传输"，而在于"释放 CPU 去做更有价值的工作"。

✅ 正确评估：DMA 的核心优势是 CPU 利用率提升，尤其在多任务环境下，释放出的 CPU 时间可以执行其他进程或处理其他 I/O 请求。

知识点 4 · 内存映射 I/O（Memory-Mapped I/O）

🎯 为什么研究这个问题

CPU 如何"对话"外部设备？要让 CPU 控制 LED 灯亮灭、从传感器读取温度、向串口发送字符，就需要某种机制让 CPU 访问设备内部的控制寄存器和数据寄存器。历史上有两种根本不同的方案：

端口映射 I/O（Port-Mapped I/O）：使用独立于内存的 I/O 地址空间和专用指令（如 x86 的 IN/OUT）。这种方式在早期 PC 架构中很常见，但它需要特殊的 I/O 指令，增加了 ISA 的复杂性。

内存映射 I/O（Memory-Mapped I/O, MMIO）：将设备寄存器映射到物理地址空间的特定区域。CPU 使用普通的 load/store 指令即可访问设备。这种方式无需特殊指令、编程模型统一，是现代架构（包括 RISC-V、ARM）的主流选择。

理解 MMIO 是编写设备驱动程序、理解嵌入式系统和进行裸机编程（bare-metal programming）的必备知识。

🔍 抽象化方法

端口映射 I/O（Port-Mapped I/O）的特点：独立的 I/O 地址空间，需要特殊指令（x86：IN AL, DX / OUT DX, AL），地址空间通常较小（x86 仅 64KB I/O 端口），只能通过汇编或特殊内联函数访问。

内存映射 I/O（MMIO）的特点：设备寄存器被分配到物理地址范围内。CPU 使用标准的 load/store 指令访问。无需特殊指令，任何语言都可以通过指针操作访问设备。

MMIO 的优势：① 不需要额外的 I/O 指令，简化了指令集设计；② 统一的地址空间使得编程模型更简洁；③ 高级语言（C/C++）可以通过指针直接操作设备寄存器；④ 可以利用虚拟内存的保护机制控制 I/O 访问权限。

MMIO 的注意事项：① 占用部分物理地址空间（减少可用内存地址范围）；② MMIO 地址绝对不能被缓存——否则 CPU 会读到缓存中的旧值，而非设备的实时状态；③ 访问可能有副作用（side effect）——读取某些设备寄存器会改变设备状态。

RISC-V 地址空间布局（典型）：RISC-V 主要使用 MMIO。地址空间被划分为：RAM 区域（存放代码和数据）、ROM/Flash 区域（存放启动代码）、I/O 设备区域（UART、GPIO、定时器、中断控制器 PLIC/CLINT 等）。每个设备占据一段连续的地址范围。

📐 公式推导 / 模型建立

地址解码（Address Decoding）示例：假设 UART 设备映射到地址 0x10000000 开始的区域：

\[ \text{UART\_TX} = \text{0x10000000},\quad \text{UART\_RX} = \text{0x10000004},\quad \text{UART\_STATUS} = \text{0x10000008} \]

C 语言中的 MMIO 编程：

#define UART_TX     (*(volatile uint32_t *)0x10000000)
#define UART_RX     (*(volatile uint32_t *)0x10000004)
#define UART_STATUS (*(volatile uint32_t *)0x10000008)

#define TX_READY 0x01  // status bit: 发送就绪

void uart_putchar(char c) {
    while (!(UART_STATUS & TX_READY))  // 轮询等待发送就绪
        ;                                // busy wait
    UART_TX = c;                         // 发送字符
}

上面的 C 代码会被编译为如下 RISC-V 汇编（简化版）：

    lui  t0, 0x10000        # t0 = 0x10000000（UART 基地址）
poll:
    lw   t1, 8(t0)          # t1 = UART_STATUS
    andi t1, t1, 0x01       # 检查 TX_READY 位
    beqz t1, poll           # 未就绪则继续轮询
    li   t2, 65             # t2 = 'A'（示例字符）
    sw   t2, 0(t0)          # 写入 UART_TX → 发送字符

地址空间分区：对于 32 位系统，总地址空间为 \(2^{32} = 4\,\text{GB}\)。需要合理分配给 RAM 和 I/O 设备：

\[ \text{可用 RAM} = 2^{32} - \sum(\text{I/O 设备映射大小}) \]

在实际 RISC-V SoC 中，I/O 设备通常映射到高地址区域（如 0x10000000 以上），低地址留给 RAM。

⚠️ MMIO 地址必须标记为不可缓存（Uncacheable）！如果 CPU 缓存了设备状态寄存器的值，它将反复读到同一个旧值，永远检测不到设备状态变化。同时，C 代码中必须使用 volatile 关键字修饰 MMIO 指针——否则编译器可能"优化"掉你的轮询循环（编译器认为同一个地址的值不会被"神秘地"改变，所以只读一次然后复用）。这是嵌入式开发中最常见、也最隐蔽的 Bug 之一。
🌐 实际应用

GPU 帧缓冲（Framebuffer）：GPU 的显存（VRAM）被映射到 CPU 的物理地址空间。CPU 通过普通的 store 指令向帧缓冲区写入像素数据，GPU 自动将其显示到屏幕上。Linux 的 /dev/fb0 就是对帧缓冲 MMIO 的抽象。

PCIe 设备 BAR（Base Address Register）：PCIe 设备通过 BAR 寄存器告诉系统它需要多大的 MMIO 地址空间。BIOS/UEFI 在启动时为每个 PCIe 设备分配 MMIO 地址范围（BAR 配置）。这就是为什么你在设备管理器中看到的"内存范围"就是该设备的 MMIO 映射区域。

ARM 外设内存映射：ARM 处理器（如树莓派的 BCM2835）将所有外设（GPIO、UART、SPI、I2C、PWM、定时器）映射到固定的地址区域。嵌入式程序员通过在 C 中直接操作这些地址来控制硬件——例如设置 GPIO 引脚的高低电平来点亮 LED。

嵌入式 GPIO 控制：在 RISC-V 嵌入式 MCU（如 SiFive FE310）中，GPIO 端口被映射到特定地址。通过写入 GPIO 的方向寄存器（设为输出）和数据寄存器（设为高/低），可以直接控制物理引脚电平。这就是裸机编程（bare-metal programming）的核心技术。

✏️ 例题分析与易错点

题目：编写 C 代码，通过 MMIO 向 UART 发送字符串 "Hello"。UART 基地址为 0x10000000，TX 数据寄存器偏移为 0x00，状态寄存器偏移为 0x08，TX_READY 位为 bit 0。

正确实现：

#define UART_BASE   0x10000000
#define UART_TX     (*(volatile uint32_t *)(UART_BASE + 0x00))
#define UART_STATUS (*(volatile uint32_t *)(UART_BASE + 0x08))
#define TX_READY    (1 << 0)

void uart_puts(const char *s) {
    while (*s) {
        while (!(UART_STATUS & TX_READY))
            ;           // 等待发送就绪
        UART_TX = *s;   // 发送一个字符
        s++;
    }
}

常见的错误版本（不使用 volatile）：

// ❌ 错误：没有 volatile！
#define UART_STATUS (*(uint32_t *)(UART_BASE + 0x08))

// 编译器 -O2 优化后可能变成：
// if (!(UART_STATUS & TX_READY))
//     while(1);  // 无限循环！
// 因为编译器认为 UART_STATUS 不会被"外部"改变

❌ 常见错误：省略 volatile 关键字。在 -O0（无优化）模式下代码可能正常工作，但一旦开启优化（-O2），编译器会缓存 MMIO 读取的值，导致轮询循环变成无限循环或完全被移除。

✅ 黄金法则：访问 MMIO 地址的指针必须用 volatile 修饰。同时，在操作系统层面，MMIO 对应的页表项必须设置为不可缓存（Uncacheable），确保硬件层面也不缓存这些地址。
