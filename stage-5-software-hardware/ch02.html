汇编与编译

知识点 1 · 从 C 到汇编——编译器做了什么

🎯 为什么研究这个问题

当你在 C 语言中写下 int x = a + b; 时，处理器硬件实际执行的是什么？处理器不认识 C 语言——它只认识由 0 和 1 组成的机器指令。在你的源代码和硬件之间，存在着一位沉默而强大的翻译官：编译器（Compiler）。

理解编译过程就是理解人类思维与机器逻辑之间的桥梁。高级语言让我们用接近自然语言的方式表达算法，而编译器将这些抽象表达忠实地翻译成处理器能直接执行的机器指令。这个翻译过程并非简单的逐行替换——编译器需要处理类型系统、作用域、控制流、内存管理等大量语义信息。

对于计算机体系结构的学习者来说，理解 C 到汇编的映射关系是打通"软硬交界"的核心技能：它让你知道每一行高级代码在硬件上的真实代价，从而写出更高效的程序，也更深刻地理解处理器的设计权衡。

🔍 抽象化方法

从 C 源代码到可执行程序，需要经过四个阶段：

① 预处理（Preprocessing）：处理以 # 开头的指令——展开 #include（插入头文件内容）、替换 #define 宏、处理 #ifdef 条件编译。输出仍是 C 源代码，但已消除所有预处理指令。

② 编译（Compilation）：将预处理后的 C 代码翻译为汇编语言（Assembly）。这是最核心的阶段——编译器进行词法分析、语法分析、语义检查、中间代码生成、优化，最终输出目标平台的汇编代码。

③ 汇编（Assembly）：汇编器（Assembler）将汇编语言中的助记符（如 add、lw）翻译为二进制机器码，生成目标文件（Object File）（.o 文件）。

④ 链接（Linking）：链接器（Linker）将多个目标文件和库组合成最终的可执行文件，解析跨文件的符号引用。

在编译阶段，C 语言的各种构造与汇编指令之间存在系统性的映射关系：算术运算对应 add/sub/mul 等指令；变量对应寄存器或内存位置；if/else 分支对应 beq/bne 等条件跳转指令；循环对应带有向后跳转的条件分支；数组访问对应基址 + 偏移量寻址。编译器还需要进行寄存器分配（Register Allocation）——将程序中的变量映射到有限的物理寄存器上。

📐 公式推导 / 模型建立

让我们用具体的例子展示 C 到汇编的映射过程。

① 算术运算：对于 int sum = a + b;（假设 a、b 的地址在 x10 指向的结构中）：

lw   x5, 0(x10)    # 从内存加载 a 到寄存器 x5
lw   x6, 4(x10)    # 从内存加载 b 到寄存器 x6
add  x7, x5, x6    # x7 = x5 + x6，即 sum = a + b
sw   x7, 8(x10)    # 将 sum 存回内存

注意：一条简单的 C 加法语句需要 4 条汇编指令——2 条 load、1 条 add、1 条 store。这就是 Load/Store 架构的特点：运算只能在寄存器间进行，必须先从内存加载数据。

② if-else 分支：对于 if (a == b) x = 1; else x = 2;：

    beq  x5, x6, equal   # 若 a == b，跳转到 equal
    addi x7, x0, 2       # else: x = 2
    j    end              # 跳过 if 分支
equal:
    addi x7, x0, 1       # if: x = 1
end:
    sw   x7, 0(x10)      # 存储结果

③ while 循环：对于 while (i < n) { sum += arr[i]; i++; }：

loop:
    bge  x5, x6, done    # 若 i >= n，退出循环
    slli x28, x5, 2      # x28 = i * 4（int 为 4 字节）
    add  x28, x10, x28   # x28 = arr 基址 + 偏移
    lw   x29, 0(x28)     # x29 = arr[i]
    add  x7, x7, x29     # sum += arr[i]
    addi x5, x5, 1       # i++
    j    loop             # 跳回循环开头
done:

从指令数量分析：循环体每次迭代执行 6 条指令。若循环执行 \(N\) 次，则总指令数为 \(6N + 1\)（最后一次 bge 跳出循环）。这类分析对性能估算至关重要。

⚠️ 编译器可能生成与你预期完全不同的汇编代码！使用 -O0（无优化）和 -O2（标准优化）编译同一段 C 代码，生成的汇编可能面目全非。永远不要假设 C 代码行与汇编指令之间存在 1:1 的对应关系——编译器可能重排指令、消除冗余计算、甚至完全移除你认为"必要"的代码。
🌐 实际应用

使用 gcc -S 查看汇编输出：在终端中运行 gcc -S -O0 main.c 会生成 main.s 文件，内容是人类可读的汇编代码。加上 -O2 可以看到优化后的版本。比较两者的差异是理解编译器行为的最佳实践。

Godbolt 编译器探索器：godbolt.org 是一个在线工具，支持几十种编译器和目标架构。你可以在左侧编写 C 代码，右侧实时看到生成的汇编，还能高亮对应关系。这是学习编译器行为的最直观工具。

GDB 汇编级调试：使用 gdb 的 disassemble 命令可以查看当前函数的反汇编代码，stepi/nexti 可以逐指令执行，info registers 查看所有寄存器的值。对于性能关键代码的调试，汇编级调试是不可或缺的技能。

理解性能关键代码：在游戏引擎、数据库内核、密码学库等对性能极度敏感的场景中，工程师需要检查编译器的汇编输出，确认热路径（Hot Path）上没有不必要的内存访问或分支，甚至手写内联汇编来榨取最后一点性能。

✏️ 例题分析与易错点

题目：将以下 C 代码翻译为 RISC-V 汇编：for(int i=0; i<n; i++) sum += arr[i];。假设 n 在 x6，arr 基址在 x10，sum 在 x7（初始为 0），i 在 x5。

逐步翻译：

    addi x5, x0, 0       # i = 0
    addi x7, x0, 0       # sum = 0
loop:
    bge  x5, x6, done    # if (i >= n) goto done
    slli x28, x5, 2      # x28 = i * 4（每个 int 占 4 字节）
    add  x28, x10, x28   # x28 = &arr[i]（基址 + 偏移）
    lw   x29, 0(x28)     # x29 = arr[i]
    add  x7, x7, x29     # sum += arr[i]
    addi x5, x5, 1       # i++
    j    loop             # 回到循环条件判断
done:

❌ 常见错误：忘记用 slli x28, x5, 2 将索引乘以元素大小！arr[i] 的地址不是 arr + i，而是 arr + i × sizeof(int)。如果直接用 add x28, x10, x5 计算地址，会导致地址偏移错误——每次只偏移 1 字节而非 4 字节，读取到错误的数据。

✅ 正确做法：始终记住数组地址计算公式：\(\text{addr}(\text{arr}[i]) = \text{base} + i \times \text{sizeof(element)}\)。对于 int 数组，需要左移 2 位（乘以 4）；对于 long（8 字节），左移 3 位。

知识点 2 · 调用约定（Calling Convention）与栈帧（Stack Frame）

🎯 为什么研究这个问题

当函数 A 调用函数 B 时，看似简单的一次 jal 跳转背后隐藏着大量必须回答的问题：参数通过什么方式传递给 B？B 的返回值放在哪里？A 正在使用的寄存器会不会被 B 覆盖？B 返回后，程序如何知道跳回 A 的哪一行继续执行？

如果没有一套统一的规则来回答这些问题，分离编译（Separate Compilation）就不可能实现——每个 .c 文件独立编译为 .o 文件，最终由链接器组合，这要求所有编译器对函数调用的"礼仪"达成一致。这套规则就是调用约定（Calling Convention）。

调用约定还决定了栈帧（Stack Frame）的布局方式——每次函数调用在栈上分配的一块内存区域，用于保存返回地址、局部变量和需要保护的寄存器。理解栈帧结构是理解递归、调试崩溃（查看 backtrace）和分析安全漏洞（缓冲区溢出攻击）的基础。

🔍 抽象化方法

RISC-V 的调用约定对 32 个通用寄存器做了明确的角色划分：

参数寄存器 a0–a7（x10–x17）：前 8 个函数参数依次放在 a0 至 a7 中。超过 8 个的参数通过栈传递。函数返回值放在 a0（如果需要 64 位返回值，使用 a0 和 a1）。

Callee-saved 寄存器 s0–s11（x8–x9, x18–x27）：被调用者（callee）负责保护。如果函数要使用这些寄存器，必须先将原值保存到栈帧中，返回前恢复。这保证了调用者（caller）在函数返回后，这些寄存器的值不变。

Caller-saved 寄存器 t0–t6（x5–x7, x28–x31）：临时寄存器，调用者自行负责。如果调用者在调用前后需要保留这些寄存器的值，必须自己保存和恢复。

特殊寄存器：ra（x1）保存返回地址；sp（x2）是栈指针；gp（x3）是全局指针；tp（x4）是线程指针。

栈帧结构：栈从高地址向低地址增长。每次函数调用，sp 向下移动分配栈帧空间。栈帧中依次存放：返回地址（ra）、保存的 callee-saved 寄存器、局部变量、为被调用函数准备的参数（如果超过 8 个）。

📐 公式推导 / 模型建立

一个标准函数的汇编代码遵循序言-主体-尾声（Prologue-Body-Epilogue）模式：

# ─── 序言（Prologue）───
func:
    addi sp, sp, -16    # 分配 16 字节栈帧（必须 16 字节对齐）
    sw   ra, 12(sp)     # 保存返回地址
    sw   s0, 8(sp)      # 保存 callee-saved 寄存器 s0

# ─── 函数主体（Body）───
    ...                  # 使用 a0-a7 读取参数
    ...                  # 可以安全使用 s0 了
    ...                  # 结果放入 a0 作为返回值

# ─── 尾声（Epilogue）───
    lw   s0, 8(sp)      # 恢复 s0 的原始值
    lw   ra, 12(sp)     # 恢复返回地址
    addi sp, sp, 16     # 释放栈帧
    ret                  # 等价于 jalr x0, ra, 0

帧指针（Frame Pointer, FP）与栈指针的关系可以用公式表示：

\[ \text{FP} = \text{SP}_{\text{entry}} \]

局部变量通过帧指针或栈指针加偏移量访问：

\[ \text{addr}(\text{local}_k) = \text{FP} - \text{offset}_k \]

栈帧大小的计算：假设需要保存 \(r\) 个寄存器（每个 4 字节），有 \(v\) 字节的局部变量，则栈帧大小为：

\[ \text{FrameSize} = \lceil (4r + v) / 16 \rceil \times 16 \]

上取整到 16 的倍数是因为 RISC-V 要求栈指针保持 16 字节对齐。

⚠️ 忘记保存 callee-saved 寄存器是最常见的汇编 Bug 之一！如果你的函数修改了 s0–s11 中的任何一个寄存器，就必须在序言中保存它、在尾声中恢复它。否则，调用者在你返回后会发现自己依赖的寄存器值被"神秘地"改变了，导致极难排查的 Bug。同样，如果你的函数调用了其他函数（即不是叶函数），就必须保存 ra——否则 ret 时会跳到错误的地址。
🌐 实际应用

递归函数的工作原理：每次递归调用都会创建一个新的栈帧，保存自己的局部变量和返回地址。递归深度为 \(d\) 时，栈上存在 \(d\) 个栈帧。这就是为什么无限递归会导致栈溢出（Stack Overflow）——栈空间是有限的，过多的帧会耗尽可用内存。

调试中的栈回溯（Backtrace）：当程序崩溃时，调试器通过沿栈帧链逐级向上追溯，可以还原完整的函数调用序列。每个栈帧中保存的 ra 和 FP 就是回溯的线索。这就是你在 GDB 中输入 bt 看到调用栈的原理。

缓冲区溢出攻击：经典的栈溢出攻击利用了栈帧布局的可预测性。如果局部变量（如数组）相邻于 ra 的保存位置，攻击者可以通过溢出数组覆盖 ra 的值，使函数返回时跳转到恶意代码。现代防御措施包括栈金丝雀（Stack Canary）、地址空间随机化（ASLR）和不可执行栈（NX bit）。

setjmp/longjmp：C 语言的非局部跳转机制（setjmp/longjmp）本质上是保存和恢复栈帧上下文（包括 SP、FP、callee-saved 寄存器和 PC），实现跨多层函数调用的"远程返回"。

✏️ 例题分析与易错点

题目：编写递归阶乘函数 int fact(int n) 的完整 RISC-V 汇编，包括栈帧管理。

C 伪代码：if (n <= 1) return 1; else return n * fact(n-1);

fact:
    addi sp, sp, -16     # 分配栈帧
    sw   ra, 12(sp)      # 保存返回地址（因为要调用 fact 自身）
    sw   a0, 8(sp)       # 保存参数 n（调用后还需要用）

    addi x5, x0, 1       # x5 = 1
    bgt  a0, x5, recurse # if (n > 1) goto recurse
    addi a0, x0, 1       # 基本情况：return 1
    j    epilog           # 跳到尾声

recurse:
    addi a0, a0, -1      # a0 = n - 1
    jal  ra, fact         # 调用 fact(n-1)，结果在 a0 中
    lw   x5, 8(sp)       # 恢复保存的 n
    mul  a0, x5, a0      # a0 = n * fact(n-1)

epilog:
    lw   ra, 12(sp)      # 恢复返回地址
    addi sp, sp, 16      # 释放栈帧
    ret                   # 返回

❌ 常见错误：在递归调用 jal ra, fact 之前忘记保存 ra。如果不保存 ra，递归调用会覆盖 ra 的值，导致最终无法返回到最初的调用者——程序会陷入无限循环或崩溃。

✅ 关键原则：凡是非叶函数（即内部调用了其他函数的函数），都必须在序言中保存 ra。叶函数不调用其他函数，ra 不会被修改，可以省略保存。

知识点 3 · 编译器优化——为什么你的"理解"可能失效

🎯 为什么研究这个问题

你是否有过这样的经历：在 Godbolt 上打开 -O2 优化，发现编译器生成的汇编代码与你的 C 源码几乎毫无相似之处？循环消失了，函数调用不见了，甚至整个代码块被一条指令替代？这就是现代编译器优化的威力。

编译器优化是一把双刃剑。一方面，它使得程序员可以用清晰、可读的方式编写代码，而不必担心性能——编译器会自动将其转换为高效的机器码。另一方面，当你需要调试优化后的代码、分析性能瓶颈、或理解安全漏洞时，不了解编译器优化就会陷入迷雾。

理解常见的编译器优化手段，不仅能帮助你写出更容易被优化的代码（而不是写出阻碍优化的代码），还能让你在阅读编译器输出时不再困惑。这是从"能写代码"到"真正理解代码在机器上如何执行"的关键一跃。

🔍 抽象化方法

编译器的优化可以分为多个层次和类别：

① 常量折叠（Constant Folding）：在编译时直接计算常量表达式。例如 int x = 3 + 5; 直接变成 int x = 8;，运行时无需任何计算。

② 死代码消除（Dead Code Elimination）：移除不会影响程序输出的代码。如果一个变量被赋值但从未被读取，赋值操作会被直接删除。

③ 公共子表达式消除（Common Subexpression Elimination, CSE）：如果同一个表达式被计算多次且中间值未改变，只计算一次并复用结果。

④ 强度削减（Strength Reduction）：将高代价操作替换为等价的低代价操作。最典型的例子：将乘以 2 的幂次替换为左移操作，因为移位在硬件中远比乘法快。

⑤ 循环展开（Loop Unrolling）：将循环体复制多次以减少分支跳转的开销，代价是代码体积增大。

⑥ 内联展开（Function Inlining）：将函数体直接插入调用处，消除函数调用的开销（保存/恢复寄存器、跳转）。

⑦ 寄存器分配（Register Allocation）：使用图着色（Graph Coloring）等算法，将尽可能多的变量分配到寄存器中，减少对内存的访问。

优化级别：-O0 不优化（方便调试）；-O1 基础优化；-O2 标准优化（推荐用于发布）；-O3 激进优化（可能增大代码体积）；-Os 优化代码大小。

📐 公式推导 / 模型建立

让我们用一个例子展示多种优化的联合作用。考虑以下 C 代码：

int x = a * 4 + a * 4;

第一步——公共子表达式消除（CSE）：表达式 a * 4 出现了两次。编译器识别后，只计算一次：

int t = a * 4;
int x = t + t;

第二步——强度削减（Strength Reduction）：将 a * 4 替换为移位操作：

int t = a << 2;   // 左移 2 位 = 乘以 4
int x = t + t;

第三步——进一步简化：t + t 等价于 t << 1，而 (a << 2) << 1 = a << 3，即 a * 8：

int x = a << 3;   // 一条移位指令解决！

原本需要 2 次乘法和 1 次加法的表达式，经过三轮优化后只需1 条移位指令。

循环展开的量化分析：一个执行 \(N\) 次的循环，每次迭代末尾有一条条件分支。若进行 \(k\) 倍展开（每次循环体复制 \(k\) 次），则分支次数从 \(N\) 减少为：

\[ \text{branches} = \lceil N / k \rceil \]

内联展开的权衡：假设某函数体有 \(B\) 条指令，被调用 \(C\) 次，每次调用的额外开销（保存寄存器、跳转等）为 \(H\) 条指令。不内联时总开销为 \(B + C \times H\) 条指令（函数体 + 调用开销），内联后为 \(B \times C\) 条指令（每个调用点都嵌入函数体）。当 \(B\) 较小而 \(C\) 不太大时，内联有利：

\[ B \times C < B + C \times H \implies B < \frac{C \times H}{C - 1} \approx H \quad (\text{当 } C \gg 1) \]

即当函数体大小小于调用开销时，内联几乎总是有利的。

⚠️ 不要试图用"聪明"的 C 语言技巧来超越编译器优化！现代编译器（GCC、LLVM/Clang）在 -O2 级别下，几乎总是能生成比手动"优化"的 C 代码更好的汇编。比如，手动用位操作写的"快速除法"不仅可读性差，而且编译器早就知道这些技巧并会自动应用。写清晰的代码，让编译器做优化。只有在剖析（profiling）确认了瓶颈后，才考虑在最关键的内循环中手写汇编。
🌐 实际应用

volatile 关键字的意义：编译器优化可能会将它认为"不必要"的内存读取消除掉——比如连续读取同一地址两次时，它可能只读一次并复用值。但如果这个地址对应的是硬件设备寄存器（Memory-Mapped I/O），每次读取的值可能不同！volatile 告诉编译器"不要对这个变量的访问进行任何优化"，每次访问都必须真实地执行。

编译器屏障（Compiler Barrier）：在多线程和操作系统内核编程中，编译器可能会重排无数据依赖的内存操作来提升性能。编译器屏障（如 GCC 的 asm volatile("" ::: "memory")）防止编译器跨屏障重排内存操作，对于保证并发正确性至关重要。

Profile-Guided Optimization (PGO)：先用典型输入运行程序并收集性能数据（哪些分支最常走、哪些函数最热），然后将这些数据反馈给编译器进行第二次编译。编译器根据真实的运行时统计来做出更优的内联、分支预测提示和代码布局决策。

Link-Time Optimization (LTO)：传统编译器只能优化单个编译单元（.c 文件）内的代码。LTO 推迟优化到链接阶段，使编译器能跨文件进行内联、死代码消除等优化，发现更多优化机会。

✏️ 例题分析与易错点

题目：以下代码在 -O0 和 -O2 下生成的汇编有何区别？

int compute(int a) {
    int x = a * 4 + a * 4;
    return x;
}

-O0 生成的汇编（示意，RISC-V）：

compute:
    addi sp, sp, -16
    sw   a0, 0(sp)       # 把 a 存到栈上
    lw   x5, 0(sp)       # 加载 a
    slli x5, x5, 2       # a * 4
    lw   x6, 0(sp)       # 再次加载 a
    slli x6, x6, 2       # a * 4（重复计算！）
    add  a0, x5, x6      # x = a*4 + a*4
    addi sp, sp, 16
    ret

-O2 生成的汇编：

compute:
    slli a0, a0, 3       # a << 3 = a * 8，一步到位！
    ret

优化后的版本只需要2 条指令（slli + ret），而未优化版本需要 8 条以上。编译器通过 CSE、强度削减和代数化简，将整个计算压缩为一条移位指令。

❌ 常见误区：认为 -O2 下代码行为会改变。标准的编译器优化保证可观测行为（Observable Behavior）不变——即程序的输入输出、volatile 访问和文件操作完全一致。优化改变的只是中间过程的执行方式。

知识点 4 · 链接（Linking）与加载（Loading）

🎯 为什么研究这个问题

真实的程序从来不是单个文件。一个中等规模的 C 项目可能包含数十个 .c 文件和数百个库函数。每个源文件都被独立编译成 .o 目标文件——但目标文件中的代码无法独立运行：main.o 调用了 utils.o 中的函数，而 utils.o 又引用了标准库中的 printf。这些跨文件的引用该如何解决？

这就是链接器（Linker）的职责——它将多个目标文件和库"拼接"在一起，解析所有的跨文件引用，最终生成一个完整的可执行文件。而加载器（Loader）则是操作系统中的组件，负责将可执行文件读入内存、设置运行环境（栈、堆、数据段），然后跳转到程序的入口点开始执行。

理解链接和加载过程，能帮助你解决那些令人困惑的编译错误（如 "undefined reference"）、理解动态库的工作原理、分析程序的内存布局，以及理解安全漏洞的利用机制。

🔍 抽象化方法

目标文件的内部结构由多个段（Section）组成，每个段存放不同类型的数据：

.text 段（代码段）：存放编译后的机器指令。通常被标记为只读+可执行。

.data 段（数据段）：存放已初始化的全局变量和静态变量。例如 int count = 42; 中的初始值 42 就存在这里。

.bss 段：存放未初始化的全局/静态变量。.bss 在文件中不占实际空间（只记录大小），加载到内存时被全部清零。名称来自历史上的 "Block Started by Symbol"。

.rodata 段（只读数据）：存放常量数据，如字符串字面量 "Hello"、const 变量等。

此外，目标文件还包含两张关键的表：符号表（Symbol Table）列出本文件定义的和引用的所有符号（函数名、全局变量名）及其地址或未定义标记；重定位表（Relocation Table）记录哪些位置的地址需要在链接时填入正确的值。

ELF（Executable and Linkable Format）是 Linux 系统上标准的目标文件和可执行文件格式。ELF 头部包含程序入口点、段表（Section Header Table）和程序头表（Program Header Table），后者告诉加载器如何将文件映射到内存。

静态链接 vs 动态链接：静态链接在编译时将所有引用的库代码复制进可执行文件；动态链接只在可执行文件中记录对共享库（.so/.dll）的引用，实际代码在运行时由动态链接器（如 ld.so）加载。

📐 公式推导 / 模型建立

链接的核心任务是符号解析（Symbol Resolution）和地址重定位（Relocation）。以具体的例子说明：

假设 main.o 中调用了 utils.o 中定义的函数 foo。在编译 main.c 时，编译器不知道 foo 的最终地址，因此生成一个带有占位符的跳转指令：

# main.o 中的机器码（编译时）
0x0010: jal ra, 0x00000000    # 地址为占位符

同时，汇编器在重定位表中记录：

{ offset: 0x0010, type: R_RISCV_JAL, symbol: "foo" }

这告诉链接器："在偏移 0x0010 处有一条 JAL 指令，它引用了符号 foo，请在链接时填入 foo 的实际地址。"

链接器合并所有 .o 文件后，确定了 foo 位于最终地址 0x00012340，于是将占位符替换为实际地址：

# 可执行文件中（链接后）
0x0010: jal ra, 0x00012340    # 已填入实际地址

各段在内存中的虚拟地址由链接器分配，遵循公式：

\[ \text{Virtual Address} = \text{Base}_{\text{section}} + \text{offset}_{\text{in\_section}} \]

静态与动态链接的体积比较：假设程序引用了一个 2 MB 的库。静态链接后，可执行文件增大约 2 MB（库代码被完整复制进来）。动态链接后，可执行文件只增加几百字节（记录库名和符号引用），但运行时需要系统上存在对应的 .so 文件。如果 10 个程序都使用同一个库，静态链接浪费 20 MB 磁盘空间，而动态链接只需一份 2 MB 的共享库。

⚠️ "Undefined reference" 错误发生在链接阶段，不是编译阶段！如果你看到 undefined reference to 'foo'，这意味着链接器在所有目标文件和库中都找不到符号 foo 的定义。常见原因：① 忘记编译包含 foo 实现的源文件；② 忘记使用 -l 选项链接所需的库（如使用 sin() 需要 -lm）；③ C++ 的名字修饰（Name Mangling）导致符号名不匹配。不要把这个错误和编译时的 "implicit declaration" 警告混淆——后者是找不到函数声明，前者是找不到函数实现。
🌐 实际应用

动态共享库（.so / .dll）：Linux 上的 .so（Shared Object）和 Windows 上的 .dll（Dynamic Link Library）是动态链接的载体。它们在运行时被动态链接器加载到进程的地址空间中。使用 ldd 命令可以查看一个可执行文件依赖哪些共享库。

位置无关代码（PIC）：共享库需要编译为位置无关代码（gcc -fPIC），即代码中不包含绝对地址，所有地址引用都通过全局偏移表（GOT, Global Offset Table）间接访问。这使得同一份库代码可以被加载到不同进程的不同虚拟地址处。

过程链接表（PLT, Procedure Linkage Table）：动态链接的函数调用通过 PLT 实现延迟绑定（Lazy Binding）——第一次调用时才解析函数的实际地址并缓存到 GOT 中，后续调用直接跳转。这避免了程序启动时一次性解析所有符号的开销。

动态链接器 ld.so：这是 Linux 中的运行时链接器，由内核在加载 ELF 可执行文件时自动调用。它负责加载所有依赖的共享库、解析符号引用、处理重定位，最后将控制权交给程序的 _start（或 main）入口点。

✏️ 例题分析与易错点

题目：有两个源文件 main.c 和 utils.c。main.c 调用了 utils.c 中定义的函数 int add(int a, int b)。描述从源码到可执行文件的完整编译链接过程。

步骤分解：

# 步骤 1：分别编译为目标文件
gcc -c main.c  -o main.o    # main.o: 引用 add（undefined）
gcc -c utils.c -o utils.o   # utils.o: 定义 add

# 步骤 2：链接
gcc main.o utils.o -o program  # 链接器解析 add 的引用

链接过程中，链接器执行以下操作：① 合并所有 .text 段为一个连续的代码区域；② 合并 .data 和 .bss 段；③ 在 main.o 的符号表中发现 add 为未定义（undefined）符号；④ 在 utils.o 的符号表中找到 add 的定义和地址；⑤ 根据重定位表，将 main.o 中所有引用 add 的占位符替换为实际地址。

❌ 常见错误 1：编译时忘记链接 math 库。使用 sin()、sqrt() 等函数时，只 #include <math.h> 不够——这只提供了声明。还需要在链接时加 -lm 选项来链接 libm 库，否则会报 undefined reference to 'sin'。

❌ 常见错误 2：链接顺序错误。链接器从左到右扫描目标文件和库，如果库出现在引用它的目标文件之前，符号可能无法解析。正确做法：将 -l 选项放在命令最后，如 gcc main.o utils.o -lm。
