Chapitre 1

GPU 架构（GPU Architecture）

知识点 1 · GPU vs CPU——吞吐量 vs 延迟的哲学分歧

🎯 为什么研究这个问题

CPU 是"全能选手"——通过复杂的控制逻辑、深流水线、分支预测和乱序执行，把单个任务的延迟（Latency）压到最低。然而，当面对数百万个相互独立的简单计算——例如屏幕上每个像素的颜色——CPU 的少量核心就力不从心了。

GPU 的设计哲学截然不同：放弃单线程性能，用大量简单核心换取极高的吞吐量（Throughput）。这不是"谁更强"的问题，而是两种根本不同的计算哲学：CPU 优化延迟，GPU 优化吞吐。理解这种分歧是理解整个 GPU 架构的起点。

🔍 抽象化方法

CPU 架构特征：复杂的控制逻辑（分支预测器、指令重排缓冲区、寄存器重命名）、深流水线（15-20+ 级）、大容量多级缓存（L1/L2/L3 可达数十 MB）、少量强核心（4-64 核）。晶体管预算中约 50% 用于缓存和控制逻辑，只有约 10-20% 直接用于 ALU。

GPU 架构特征：简单的控制逻辑（无复杂分支预测、无乱序执行）、小缓存、大量简单核心（数千到上万个 ALU）。晶体管预算中约 80% 用于 ALU，控制逻辑和缓存占比极小。

类比：CPU 像一辆跑车——速度快但只坐 2 人；GPU 像一列火车——单程慢但一次运数千人。当乘客数量巨大时，火车的总运力远超跑车。

📐 公式推导 / 模型建立

Amdahl 定律（Amdahl's Law）量化了并行加速的极限。设程序中可并行化的比例为 \(P\)，使用 \(N\) 个并行处理单元，最大加速比为：

\[ S = \frac{1}{(1 - P) + \dfrac{P}{N}} \]

当 \(N \to \infty\) 时，\(S \to \frac{1}{1 - P}\)。即使并行部分无限加速，串行部分决定了加速比的上限。

GPU 适用场景：\(P \to 1\)。像素着色、矩阵乘法、卷积等任务的并行比例极高（\(P > 0.99\)），此时 GPU 的数千核心可发挥巨大优势。例如 \(P = 0.99, N = 10000\) 时，\(S \approx 99.0\)。

CPU 适用场景：\(P\) 较小。当程序包含大量串行逻辑、复杂分支和数据依赖时，增加核心数收效甚微。例如 \(P = 0.5, N = 10000\) 时，\(S \approx 2.0\)——几乎无法利用并行性。

⚠️ GPU 并非总是比 CPU 快！对于串行代码、分支密集型代码（如编译器、操作系统内核），CPU 的单线程性能远超 GPU。GPU 只在高度并行的数据密集型计算中才有优势。盲目将所有计算转移到 GPU 反而可能更慢（数据传输开销、低占用率）。
🌐 实际应用

游戏图形渲染：一帧 1920×1080 的画面有 207 万个像素，每个像素的着色计算基本独立（\(P \approx 1\)），是 GPU 的理想工作负载。现代 GPU 可以在 1/60 秒内完成所有像素的光照、纹理和后处理计算。

深度学习训练：神经网络的核心运算是矩阵乘法 \(C = A \times B\)。一个 \(m \times k\) 乘 \(k \times n\) 的矩阵乘法包含 \(m \times n \times k\) 次独立的乘加运算，完美适配 GPU 的大规模并行。NVIDIA A100 GPU 的 Tensor Core 专门为此优化，训练速度比 CPU 快 10-100 倍。

加密货币挖矿：哈希计算（如 SHA-256）是高度并行的——可以同时尝试数十亿个不同的 nonce 值。GPU 的大规模并行能力使其成为早期加密货币挖矿的主力（后被 ASIC 取代）。

✏️ 例题分析与易错点

题目：某程序 95% 可并行化（\(P = 0.95\)），用一块 1024 核 GPU 加速。(a) 求最大加速比；(b) 若要达到 15 倍加速，至少需要多少并行核心？

解答：

(a) \(S = \frac{1}{0.05 + 0.95/1024} = \frac{1}{0.05 + 0.000928} = \frac{1}{0.050928} \approx 19.6\)

(b) 令 \(\frac{1}{0.05 + 0.95/N} = 15\)，解得 \(0.05 + 0.95/N = 1/15\)，\(0.95/N = 1/15 - 1/20 = 1/60\)，\(N = 57\)。

仅需 57 个核心就能达到 15 倍加速，而从 57 核增加到 1024 核只额外获得约 4.6 倍的提升——边际递减效应非常明显。

❌ 常见错误：认为 1024 核能获得 1024 倍加速，忽略了 5% 的串行部分限制了上限为 \(1/(1-0.95) = 20\) 倍。

✅ 正确理解：串行比例 \(1-P\) 是并行加速的"天花板"，增加更多核心无法突破这个极限。

⚡ 快速检测
程序中 90% 可并行化（P=0.9），使用无限多处理器，理论最大加速比是？

知识点 2 · SIMT（Single Instruction Multiple Threads）模型

🎯 为什么研究这个问题

GPU 拥有数千个 ALU，但如何调度这些 ALU 执行成千上万的线程？如果每个线程都有独立的指令获取和解码单元，硬件成本会爆炸。CPU 的 SIMD（Single Instruction Multiple Data）扩展（如 AVX-512）可以用一条指令处理多个数据，但程序员必须显式管理数据打包和向量宽度——编程难度很大。

NVIDIA 提出的 SIMT（Single Instruction Multiple Threads）模型是一种优雅的折中：硬件以线程为抽象单位（每个线程有自己的寄存器和逻辑程序计数器），但在执行时将多个线程捆绑为 Warp，共享同一条指令——兼顾了编程灵活性和硬件效率。

🔍 抽象化方法

Warp（线程束）：NVIDIA 架构中，32 个线程组成一个 Warp。同一个 Warp 中的线程在同一时钟周期执行相同的指令，但操作各自的数据（不同的寄存器内容）。AMD 的等效概念是 Wavefront（GCN 架构为 64 线程，RDNA 架构为 32 线程）。

线程的独立性：每个线程有自己的寄存器组和逻辑程序计数器。从程序员视角，每个线程都是独立的——可以执行不同的分支路径。但硬件上，Warp 内只有一个物理指令获取/解码单元。

与 SIMD 的区别：SIMD 要求数据在连续的向量寄存器中对齐，分支处理需要程序员手动用掩码管理。SIMT 让线程"看起来"独立，分支处理由硬件自动通过谓词执行（Predication）完成——编程模型更简单，但分支发散（Divergence）仍有性能代价。

📐 公式推导 / 模型建立

衡量 GPU 利用率的关键指标是占用率（Occupancy）：

\[ \text{Occupancy} = \frac{\text{Active Warps per SM}}{\text{Max Warps per SM}} \]

占用率受三个因素制约：

① 寄存器限制：若 SM 有 \(R_{\text{total}}\) 个寄存器，每线程用 \(r\) 个寄存器，则最多驻留 \(\lfloor R_{\text{total}} / (r \times 32) \rfloor\) 个 Warp（每 Warp 32 线程）。

② 共享内存限制：若 SM 有 \(S_{\text{total}}\) 字节共享内存，每 Block 用 \(s\) 字节，则 SM 最多运行 \(\lfloor S_{\text{total}} / s \rfloor\) 个 Block。

③ Warp 槽位限制：SM 有硬件上限（如 Ampere 架构最多 48 Warp/SM）。

实际占用率 = min(三个限制) / Max Warps per SM。高占用率有助于隐藏内存延迟（当一个 Warp 等待数据时，调度器切换到其他就绪的 Warp）。

⚠️ 分支发散（Branch Divergence）是 GPU 性能的头号杀手。当 Warp 内的线程走不同的 if/else 分支时，硬件必须序列化两条路径——先执行 if 路径（屏蔽 else 线程），再执行 else 路径（屏蔽 if 线程）。一个二分支发散让 SIMT 效率降至 50%；多层嵌套分支可能导致指数级性能下降。
🌐 实际应用

分支发散的处理：当 Warp 内线程遇到条件分支：(1) 如果所有线程走同一路径（Uniform Branch），无性能损失；(2) 如果不同线程走不同路径（Divergent Branch），硬件通过活跃掩码（Active Mask）屏蔽非活跃线程，依次执行每条路径，然后在汇合点（Reconvergence Point）合并。

CUDA 编程中的优化策略：将相邻线程映射到相似的数据/条件，使同一 Warp 内的线程尽可能走相同分支。例如对图像处理，让 Warp 内 32 个线程处理空间相邻的 32 个像素（它们通常有相似的亮度/颜色条件）。

Volta 架构的独立线程调度：从 Volta（2017）起，NVIDIA 引入了独立线程调度（Independent Thread Scheduling），每个线程有真正的独立程序计数器，允许更灵活的同步和发散处理。但这也带来了新的正确性问题（如隐式 Warp 同步不再安全）。

✏️ 例题分析与易错点

题目：某 GPU SM 有 65536 个寄存器，最多支持 48 个 Warp，共享内存 96 KB。一个内核每线程用 48 个寄存器，每 Block 有 256 个线程，使用 32 KB 共享内存。求占用率。

解答：

① 寄存器限制：每 Warp 用 \(48 \times 32 = 1536\) 个寄存器。最多 \(\lfloor 65536/1536 \rfloor = 42\) 个 Warp。

② 共享内存限制：每 Block 用 32 KB，SM 有 96 KB → 最多 3 个 Block。每 Block 有 \(256/32 = 8\) 个 Warp → 最多 \(3 \times 8 = 24\) 个 Warp。

③ 硬件限制：48 个 Warp。

实际：\(\min(42, 24, 48) = 24\)。占用率 = \(24/48 = 50\%\)。共享内存是瓶颈。

❌ 常见错误：只考虑寄存器限制而忽略共享内存和 Warp 槽位限制。

✅ 正确做法：三个限制都要算，取最小值。使用 NVIDIA Occupancy Calculator 可自动分析。

⚡ 快速检测
一个 Warp 的 32 个线程遇到 if-else，其中 8 个走 if、24 个走 else。SIMT 效率是？

知识点 3 · 流处理器（Streaming Multiprocessor / Compute Unit）

🎯 为什么研究这个问题

如果说 GPU 是一座工厂，那么 SM（Streaming Multiprocessor，流处理器）就是工厂中的一条生产线。GPU 的全部计算能力由多个 SM 组成（NVIDIA 称 SM，AMD 称 CU = Compute Unit）。

理解 SM 的内部结构——它有哪些功能单元、如何调度 Warp、如何管理寄存器和共享内存——是理解 GPU 性能调优的核心。SM 是 GPU 架构中最重要的中间抽象层：它连接了上层的编程模型（线程、Block）和下层的硬件执行（ALU、存储）。

🔍 抽象化方法

一个典型的 NVIDIA Ampere SM 包含以下组件：

① Warp 调度器（Warp Scheduler）：4 个调度器，每周期可以从就绪 Warp 池中选择 4 个 Warp 发射指令。当 Warp 因内存访问而停顿时，调度器立即切换到其他就绪 Warp——这就是延迟隐藏（Latency Hiding）的核心机制。

② 寄存器文件（Register File）：64K × 32-bit 寄存器（256 KB），是 SM 中最大的存储结构。寄存器按 Warp 静态分配，不存在寄存器重命名——分配多少就占多少，直到 Warp 退出。

③ 计算单元：FP32 ALU（64 个）、FP64 ALU（32 个）、INT32 ALU（64 个）、Tensor Core（4 个第三代）、LD/ST 单元（32 个）、SFU（Special Function Unit，16 个，执行 sin/cos/exp 等）。

④ 共享内存/L1 缓存：Ampere SM 有 192 KB 可配置的 Shared Memory/L1 Cache。程序可以选择划分比例（如 128 KB Shared + 64 KB L1）。

📐 公式推导 / 模型建立

理论峰值 FLOPS 计算：

\[ \text{Peak FLOPS} = N_{\text{cores}} \times f_{\text{clock}} \times 2 \quad (\text{FMA counts as 2 ops}) \]

其中 \(N_{\text{cores}}\) 是 FP32 CUDA 核心总数，\(f_{\text{clock}}\) 是 Boost 频率，\(\times 2\) 是因为 FMA（Fused Multiply-Add）\(a \times b + c\) 在一个周期内完成，算作 2 次浮点运算。

实例：NVIDIA RTX 4090

\[ \text{Peak FP32} = 16384 \times 2.52 \times 10^9 \times 2 = 82.6 \text{ TFLOPS} \]

实例：NVIDIA A100

\[ \text{Peak FP32} = 6912 \times 1.41 \times 10^9 \times 2 = 19.5 \text{ TFLOPS} \]

A100 的 FP32 核心数少于 RTX 4090，但其 Tensor Core 在 FP16/BF16 矩阵运算中可达 312 TFLOPS——这是独立的数据通路，不计入常规 FLOPS。

⚠️ Tensor Core 的算力不应与常规 FLOPS 混为一谈。Tensor Core 执行的是 \(D = A \times B + C\) 的小矩阵运算（如 4×4 FP16），有独立的数据通路和精度要求。只有使用了 WMMA（Warp Matrix Multiply-Accumulate）指令或特定库（如 cuBLAS、cuDNN）的代码才能利用 Tensor Core。普通 CUDA 代码只能使用常规 FP32/FP64 单元。
🌐 实际应用

NVIDIA Ampere SM vs AMD RDNA3 CU 对比：

Ampere SM：4 个 Warp 调度器，128 个 FP32 ALU（或 64 FP32 + 64 INT32 同时），4 个 Tensor Core，192 KB 共享缓存。

RDNA3 CU：2 个 SIMD 单元（各 32 宽），64 个 FP32 ALU，2 个 AI 加速器（矩阵运算），128 KB 共享缓存。RDNA3 引入了 WMMA 指令支持，类似 NVIDIA 的 Tensor Core。

延迟隐藏机制：GPU 内存延迟约 400-800 个时钟周期（远高于 CPU 的 L1 命中 4 周期）。SM 通过在数十个 Warp 之间快速切换来隐藏这个延迟——当 Warp A 等待数据时，调度器立即执行 Warp B 的指令。这就是为什么高占用率如此重要：更多的驻留 Warp = 更多可切换的对象 = 更好的延迟隐藏。

寄存器压力（Register Pressure）：每个线程使用的寄存器越多，SM 能同时驻留的 Warp 越少，占用率越低。编译器优化选项（如 --maxrregcount）可以限制寄存器使用量，但可能导致寄存器溢出（Register Spill）到较慢的本地内存。

✏️ 例题分析与易错点

题目：某 GPU 有 80 个 SM，每个 SM 有 128 个 FP32 核心，Boost 频率 1.5 GHz。求 (a) 理论峰值 FP32 FLOPS；(b) 若改用 Tensor Core（每 SM 有 4 个 Tensor Core，每个每周期完成 256 次 FP16 FMA），Tensor Core 的峰值 FP16 算力。

解答：

(a) 常规 FP32：\(80 \times 128 \times 1.5 \times 10^9 \times 2 = 30.72 \text{ TFLOPS}\)

(b) Tensor Core FP16：每 SM 每周期 \(4 \times 256 \times 2 = 2048\) 次 FP16 运算。总计 \(80 \times 2048 \times 1.5 \times 10^9 = 245.8 \text{ TFLOPS}\)

Tensor Core 算力是常规 FP32 的约 8 倍——这正是深度学习训练大量使用混合精度（FP16 + FP32）的原因。

❌ 常见错误：把 Tensor Core FLOPS 和常规 FLOPS 简单相加作为 GPU 总算力。

✅ 正确理解：两者是独立的数据通路，实际能用多少取决于代码是否调用了对应的指令。

知识点 4 · 显存（VRAM）与带宽

🎯 为什么研究这个问题

GPU 拥有数千个 ALU，每周期可以执行数万次浮点运算——但如果数据供应不上，ALU 就只能空转等待。这就是"算力-带宽"矛盾：GPU 的计算能力增长速度远超内存带宽增长速度。

显存带宽（Memory Bandwidth）是决定 GPU 实际性能的关键因素。很多工作负载（如大规模矩阵运算、大模型推理）并非"计算瓶颈"而是"内存瓶颈"。理解显存技术（GDDR6/GDDR6X/HBM）和Roofline 模型，才能准确判断性能瓶颈并有针对性地优化。

🔍 抽象化方法

GDDR6：常见游戏 GPU 显存。使用较窄的总线（128/192/256-bit），通过高时钟频率（16-18 Gbps 数据率）实现带宽。成本相对较低。例如 RTX 3060 使用 192-bit GDDR6，带宽 360 GB/s。

GDDR6X：NVIDIA 独家使用的改进版，采用 PAM4（4 级脉冲幅度调制）编码——每个信号周期传输 2 bit 而非 1 bit，有效翻倍数据率。RTX 4090 使用 384-bit GDDR6X @ 21 Gbps，带宽 1008 GB/s。

HBM2e / HBM3：高带宽内存（High Bandwidth Memory），通过 TSV（硅通孔，Through-Silicon Via）技术在中介层（Interposer）上垂直堆叠多层 DRAM Die。使用极宽总线（1024-bit 每堆栈 × 多堆栈），以较低时钟频率实现极高带宽。NVIDIA A100 使用 5 堆栈 HBM2e，总带宽 2039 GB/s。代价：成本高、需特殊封装。

为什么 AI 用 HBM，游戏用 GDDR？AI 训练/推理的工作集（模型参数 + 激活值）远大于游戏帧缓冲区，且访问模式更偏向大规模顺序读取，HBM 的高带宽优势更明显。游戏 GPU 的成本敏感性更高，GDDR 的性价比更优。

📐 公式推导 / 模型建立

带宽计算公式：

\[ BW = \text{bus\_width} \times \text{data\_rate} \times \text{channels} \]

例如 RTX 4090：\(BW = 384 \text{ bit} \times 21 \times 10^9 / 8 = 1008 \text{ GB/s}\)

Roofline 模型是分析 GPU 性能的核心工具。它将工作负载的性能建模为：

\[ \text{Performance} = \min(\text{Peak\_FLOPS},\ BW \times AI) \]

其中 AI（Arithmetic Intensity，算术强度）= FLOP / Byte，即每字节内存访问执行的浮点运算数。

拐点（Ridge Point）：当 \(AI = \text{Peak\_FLOPS} / BW\) 时，计算和带宽恰好平衡。

• 若 \(AI < \text{拐点}\)：内存瓶颈（Memory-bound），增加带宽有效

• 若 \(AI > \text{拐点}\)：计算瓶颈（Compute-bound），增加 ALU 有效

A100 的拐点：\(19.5 \text{ TFLOPS} / 2039 \text{ GB/s} \approx 9.6 \text{ FLOP/Byte}\)。大多数深度学习推理的 AI 在 1-10 范围，因此是内存瓶颈——这解释了为什么 AI 芯片如此看重 HBM 带宽。

⚠️ 广告中的带宽是理论峰值，实际有效带宽可能只有 60-80%。原因包括：(1) 非合并访问（Non-coalesced Access）——Warp 中线程访问不连续的地址，导致多次内存事务；(2) Bank Conflict——共享内存中多个线程访问同一 Bank；(3) 对齐问题——访问地址不对齐到缓存行边界。优化内存访问模式（特别是内存合并 Memory Coalescing）是 GPU 性能优化的基本功。
🌐 实际应用

NVIDIA A100（数据中心 AI）：HBM2e，5 堆栈 × 1024-bit，80 GB 容量，2039 GB/s 带宽。单卡可容纳 GPT-3 175B 参数的一部分（FP16 约 350 GB，需多卡）。HBM 的高带宽确保 Tensor Core 不会"饿死"。

RTX 4090（游戏/创作）：GDDR6X，384-bit 总线，24 GB 容量，1008 GB/s 带宽。对于 4K 游戏，帧缓冲区约 100-200 MB，纹理数据通过流式加载管理，GDDR6X 带宽足够。

Roofline 模型实践：矩阵乘法 \(C = A \times B\)（\(M \times K\) 乘 \(K \times N\)）的算术强度约为 \(\frac{2MKN}{(MK+KN+MN) \times \text{bytes}}\)。当矩阵足够大时 AI 很高（计算瓶颈），适合利用 Tensor Core。当矩阵小时 AI 低（内存瓶颈），需要算子融合（Kernel Fusion）来减少内存访问。

✏️ 例题分析与易错点

题目：某 GPU 峰值 FP32 算力 40 TFLOPS，显存带宽 800 GB/s。工作负载每读取 4 字节（1 个 FP32）执行 8 次浮点运算。判断该工作负载是计算瓶颈还是内存瓶颈，并计算实际性能。

解答：

算术强度：\(AI = 8 / 4 = 2 \text{ FLOP/Byte}\)

拐点：\(40 \times 10^{12} / (800 \times 10^9) = 50 \text{ FLOP/Byte}\)

\(AI = 2 < 50 = \text{拐点}\)，因此是内存瓶颈。

实际性能：\(\min(40\text{T}, 800\text{G} \times 2) = \min(40\text{T}, 1.6\text{T}) = 1.6 \text{ TFLOPS}\)

实际只利用了峰值算力的 \(1.6/40 = 4\%\)——这就是为什么许多 GPU 程序看似"算力很强"却远达不到峰值。

❌ 常见错误：认为"GPU 有 40 TFLOPS 就一定能跑到 40 TFLOPS"，忽略了带宽限制。

✅ 正确做法：先用 Roofline 模型判断瓶颈类型，再针对性优化（内存瓶颈→提高数据复用率/合并访问；计算瓶颈→使用 Tensor Core/减少冗余计算）。
