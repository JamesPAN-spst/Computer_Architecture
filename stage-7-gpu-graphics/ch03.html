Chapitre 3

着色器（Shader）与并行计算

知识点 1 · GLSL / HLSL 基础结构

🎯 为什么研究这个问题

早期 GPU 的渲染管线是完全固定的——开发者只能调整参数，无法改变光照计算逻辑。随着图形需求日益复杂（动态光影、风格化渲染、物理模拟），固定管线无法满足创意需求。

着色器（Shader）使 GPU 管线变为可编程的：开发者编写运行在 GPU 上的小程序，控制每个顶点的位置和每个像素的颜色。问题是——GPU 拥有数千个核心，着色器代码如何在这些核心上并行执行？理解着色器语言的结构，是掌握 GPU 编程的第一步。

🔍 抽象化方法

着色器按管线阶段分为多种类型：

① 顶点着色器（Vertex Shader）：对每个顶点执行，负责坐标变换（MVP）、法线变换等。输入：顶点属性（位置、法线、UV）；输出：裁剪空间坐标和插值变量。

② 片段着色器（Fragment Shader）：对每个像素片段执行，负责计算最终颜色。输入：从顶点着色器插值得到的数据；输出：像素颜色和深度。

③ 几何着色器（Geometry Shader）：可选阶段，对整个图元（三角形/线段/点）操作，可以增删顶点、生成新图元。

④ 曲面细分着色器（Tessellation Shader）：将粗糙网格动态细分为更精细的几何体，由 Hull Shader 和 Domain Shader 两部分组成。

⑤ 计算着色器（Compute Shader）：脱离图形管线，用于通用并行计算（物理模拟、后处理等）。

GLSL（OpenGL Shading Language）的基本结构：

uniform — 全局常量，由 CPU 设置，在整个 Draw Call 期间不变（如变换矩阵、光源位置）。in / out — 每顶点或每片段的输入输出数据。main() — 着色器入口函数。

HLSL（High-Level Shading Language）使用类似结构但语法不同：cbuffer 替代 uniform block，SV_POSITION 等语义（Semantics）标注输出用途。

📐 公式推导 / 模型建立

着色器编译管线将高级代码变为 GPU 可执行的机器码，过程如下：

① 源码 → 中间表示（IR）：GLSL 通过 glslangValidator 编译为 SPIR-V（Vulkan/OpenGL 4.6）；HLSL 通过 DXC 编译为 DXIL（DirectX 12）。WGSL（WebGPU Shading Language）编译为 SPIR-V 或平台特定 IR。

② IR → GPU ISA：GPU 驱动程序中的 JIT（Just-In-Time）编译器将 SPIR-V/DXIL 翻译为具体 GPU 架构的指令集（如 NVIDIA PTX → SASS，AMD GCN ISA）。

编译过程中的关键优化：

\[ \text{寄存器分配} \rightarrow \text{指令调度} \rightarrow \text{循环展开} \rightarrow \text{死代码消除} \]

寄存器用量直接影响 GPU 占用率。若每线程使用 \(R\) 个寄存器，SM 有 \(R_{\max}\) 个寄存器，则每 SM 最多容纳 \(\lfloor R_{\max} / R \rfloor\) 个线程。编译器必须在寄存器用量和指令级并行之间权衡。

⚠️ uniform 变量在整个 Draw Call 期间保持不变——不能在着色器内部修改它，也不能用它传递逐顶点或逐像素不同的数据。初学者常犯的错误是试图在 uniform 中存储随顶点变化的信息，导致所有顶点看到相同的值。
🌐 实际应用

WebGL / WebGPU：Web 端 3D 渲染使用 GLSL ES（WebGL）或 WGSL（WebGPU）编写着色器。Three.js、Babylon.js 等框架在底层管理着色器编译和参数绑定。

Pipeline State Object（PSO）预编译：现代 API（Vulkan、DX12、Metal）要求提前编译着色器并创建 PSO，将着色器状态、混合模式、渲染目标格式打包为不可变对象。这消除了运行时编译导致的卡顿（shader stutter），代价是启动时间更长。

开发期热重载（Hot Reload）：工具如 ShaderToy、Unity Shader Graph 支持实时修改着色器并立即看到效果，极大提升迭代速度。底层原理是监控文件变化后重新编译着色器并替换 PSO。

跨平台着色器方案：SPIRV-Cross 可将 SPIR-V 反编译为 GLSL、HLSL、MSL（Metal），实现"一次编写，多平台运行"。

✏️ 例题分析与易错点

题目：阅读以下 GLSL 顶点+片段着色器对，标注每个部分的作用。

顶点着色器：

#version 330 core
layout(location=0) in vec3 aPos;    // ← 顶点位置属性（CPU传入）
layout(location=1) in vec3 aColor;  // ← 顶点颜色属性
uniform mat4 uMVP;                  // ← MVP矩阵（Draw Call级常量）
out vec3 vColor;                    // ← 传递给片段着色器的插值变量
void main() {
    gl_Position = uMVP * vec4(aPos, 1.0);  // 变换到裁剪空间
    vColor = aColor;                        // 传递颜色
}

片段着色器：

#version 330 core
in vec3 vColor;              // ← 从顶点着色器插值而来
out vec4 FragColor;          // ← 最终像素颜色输出
void main() {
    FragColor = vec4(vColor, 1.0);  // RGB + Alpha=1.0
}

关键注解：in 变量在顶点着色器中逐顶点不同；光栅化器在三角形内部自动对 out/in 变量进行透视校正插值；uniform 在整个 Draw Call 中恒定。

❌ 常见错误：在片段着色器中直接使用 aPos（顶点属性），但它只对顶点着色器可见。

✅ 正确做法：需要在片段着色器中使用的数据，必须通过顶点着色器的 out → 片段着色器的 in 传递。

⚡ 快速检测
在 GLSL 中，哪种变量类型用于传递 CPU 端设置的、整个 Draw Call 不变的数据？

知识点 2 · Warp / Wavefront 与分支分歧（Divergence）

🎯 为什么研究这个问题

GPU 采用 SIMT（Single Instruction, Multiple Threads）架构：一组线程（NVIDIA 称为 Warp，32 线程；AMD 称为 Wavefront，32 或 64 线程）共享同一条指令，在不同数据上并行执行。这是 GPU 高吞吐量的核心机制。

但真实程序中必然有 if/else 分支。当同一个 Warp 内的 32 个线程在分支条件上做出不同选择时——一些走 if 路径，另一些走 else 路径——会发生什么？这就是分支分歧（Branch Divergence）问题，它是 GPU 性能优化中最关键的概念之一。

🔍 抽象化方法

当一个 Warp 遇到分支分歧时，GPU 的处理方式如下：

① 两条路径串行执行：GPU 没有能力让 Warp 内的不同线程同时执行不同指令。因此，它先执行 if 路径，再执行 else 路径（或反之）。

② 活跃掩码（Active Mask）：在执行每条路径时，不属于该路径的线程被掩码屏蔽（masked off）——它们的执行单元被禁用，不会写入结果，但仍然"占座"。

③ 重新汇合（Reconvergence）：两条路径执行完毕后，所有 32 个线程在汇合点重新同步，恢复全部活跃。

关键后果：分歧 Warp 的执行时间 = 两条路径时间之和，而非取最大值。在最坏情况下（50/50 分裂），执行时间翻倍。

📐 公式推导 / 模型建立

设 Warp 内 32 个线程中，比例 \(P\) 进入分支 A（耗时 \(C_A\) 周期），比例 \((1-P)\) 进入分支 B（耗时 \(C_B\) 周期）。

无分歧时（所有线程走同一路径）：

\[ T_{\text{no-div}} = \max(C_A, C_B) \]

有分歧时（两条路径都必须执行）：

\[ T_{\text{div}} = C_A + C_B \]

分歧代价比：

\[ \text{Slowdown} = \frac{C_A + C_B}{\max(C_A, C_B)} \]

当 \(C_A = C_B = C\) 时，\(\text{Slowdown} = 2\)——执行时间恰好翻倍。即使只有一个线程走不同路径，整个 Warp 都要付出两条路径的代价。

对于嵌套分支，代价可能更高。若有 \(k\) 层嵌套、每层二分，最坏情况下执行时间为所有 \(2^k\) 条路径之和。

⚠️ 即使 Warp 中仅有 1 个线程走了不同路径，所有 32 个线程都必须执行两条路径。分歧代价与"分歧线程数"无关，只要存在分歧就付出全部开销。因此 GPU 代码中应尽量避免线程间的条件分支。
🌐 实际应用

避免线程依赖分支：将 if(threadIdx.x < threshold) 替换为数学运算（如 step()、mix()），使所有线程执行相同指令。

短分支使用谓词化（Predication）：对于简短的 if/else（1-2 条指令），GPU 自动使用谓词寄存器：两条路径都执行，但只有满足条件的线程写入结果。开销 = 两条路径之和，但无分支指令开销。

工作排序减少分歧：在渲染中，按材质（Material）分批渲染可以确保同一 Warp 内的片段执行相同的材质着色器，避免因不同材质触发不同分支。这就是材质批处理（Material Batching）。

分支粒度对齐 Warp：如果分支边界恰好对齐 Warp 边界（例如 Warp 0-15 全走 A，Warp 16-31 全走 B），则不会产生分歧。将工作按 Warp 大小对齐是常见优化策略。

✏️ 例题分析与易错点

题目：一个内核包含以下代码，Warp 大小为 32：

if (threadIdx.x < 16) {
    A();  // 耗时 20 周期
} else {
    B();  // 耗时 10 周期
}

计算分歧版本与非分歧版本的执行时间差异。

解答：

分歧版本：线程 0-15 走 A，线程 16-31 走 B。一个 Warp 必须执行两条路径：

\[ T_{\text{div}} = 20 + 10 = 30 \text{ 周期} \]

非分歧版本（所有线程走同一路径）：

\[ T_{\text{no-div}} = \max(20, 10) = 20 \text{ 周期} \]

减速比 = \(30 / 20 = 1.5\times\)，性能损失 50%。

❌ 常见错误：认为"16 个线程走 A、16 个走 B"所以总时间 = \(\max(20,10)=20\) 周期（误以为两路径并行执行）。

✅ 正确理解：SIMT 架构下分歧路径是串行执行的，总时间是两路径之和。

⚡ 快速检测
一个 Warp 中 31 个线程走 if 路径（10 周期），1 个线程走 else 路径（5 周期）。总执行时间是？

知识点 3 · 共享内存（Shared Memory）与同步

🎯 为什么研究这个问题

GPU 的全局内存（VRAM/Global Memory）容量大但延迟高——典型的 GDDR6/HBM 访问延迟约为 400-800 个时钟周期。虽然 GPU 通过大量线程切换来隐藏延迟，但如果同一个线程块内的多个线程需要反复读取相同的数据（如矩阵乘法中的共享 tile），每次都从全局内存取数据将严重浪费带宽。

共享内存（Shared Memory）是 GPU 内每个 SM（流多处理器）上的一块片上 SRAM，容量较小（~48-164 KB）但延迟极低（~1-2 周期），允许线程块内的线程高效共享数据。理解共享内存的使用和同步机制，是 GPU 性能优化的核心。

🔍 抽象化方法

共享内存的物理组织：

共享内存被划分为 32 个 Bank（对应 Warp 的 32 个线程）。连续的 4 字节（32 位）字被交替分配到不同 Bank：地址 0-3 在 Bank 0，地址 4-7 在 Bank 1，……地址 124-127 在 Bank 31，地址 128-131 回到 Bank 0。

Bank Conflict：当同一个时钟周期内，Warp 中的多个线程访问同一个 Bank 的不同地址时，访问被串行化。若 \(k\) 个线程冲突，访问时间变为 \(k\) 倍。但如果多个线程访问同一个 Bank 的同一地址，硬件会执行广播（Broadcast），不产生冲突。

同步机制：__syncthreads()（CUDA）/ barrier()（OpenCL/GLSL）是块级屏障——线程块内所有线程都到达该点后才能继续执行。这确保数据在共享内存中写入完成后再被其他线程读取。

📐 公式推导 / 模型建立

Bank Conflict 分析：线程 \(t\) 访问地址 \(\text{addr}(t)\)，对应的 Bank 编号为：

\[ \text{bank}(t) = \left\lfloor \frac{\text{addr}(t)}{4} \right\rfloor \bmod 32 \]

若步长 stride 恰好等于 Bank 数量（32），则所有线程访问同一 Bank，产生 32-way 冲突！

Padding 技术：为避免 stride=32 的冲突，在每行末尾添加 1 个元素的 padding：

\[ \text{声明: } \texttt{__shared__ float tile[32][32+1];} \]

这使得实际 stride = 33，不再是 32 的倍数，冲突消除。

矩阵乘法的内存访问优化：朴素矩阵乘法 \(C = A \times B\)，计算 \(C[i][j]\) 需读取 A 的第 \(i\) 行和 B 的第 \(j\) 列（各 \(N\) 个元素）。总全局内存访问量 \(= 2N^3\)。

使用 \(T \times T\) 大小的 tile 加载到共享内存：每个 tile 从全局内存读取一次，被 \(T\) 个线程复用。全局内存访问减少为：

\[ \text{访问量}_{\text{tiled}} = \frac{2N^3}{T} \]

加速比 = \(T\)。典型 \(T=32\) 时，全局内存访问减少 32 倍。

⚠️ __syncthreads() 是块级屏障——它要求块内所有线程都到达同步点。如果将它放在 if 分支内，部分线程永远不会执行到同步点，导致已到达的线程无限等待——这就是死锁（Deadlock）。永远确保 __syncthreads() 在所有线程都能到达的代码路径上！
🌐 实际应用

矩阵乘法 Tiling：cuBLAS 和 cuDNN 底层大量使用共享内存 tiling。例如计算 1024×1024 矩阵乘法，tile 大小 32×32：先将 A 和 B 的子块加载到共享内存，在共享内存中完成局部乘累加，再写回全局内存。这将全局内存访问从 ~20 亿次减少到 ~6000 万次。

并行归约（Parallel Reduction）：计算数组总和时，使用共享内存存储中间结果。每一轮将数据量减半（线程 i 将 s[i] 和 s[i+stride] 相加），\(\log_2 N\) 轮后得到最终结果。每轮之间使用 __syncthreads() 同步。

直方图计算：使用共享内存中的私有直方图避免全局内存的原子操作冲突，最后再合并到全局直方图。

卷积运算：将卷积核和输入 tile（包含 halo 区域）加载到共享内存，避免重复读取重叠区域的全局内存。

✏️ 例题分析与易错点

题目：两个 \(N \times N\) 矩阵相乘（\(N = 1024\)），比较朴素算法与 tile 大小 \(T = 32\) 的共享内存版本的全局内存访问量。

解答：

朴素版本：计算 \(C[i][j]\) 需读取 A 的一行（\(N\) 次）和 B 的一列（\(N\) 次）。共 \(N^2\) 个输出元素：

\[ \text{Accesses}_{\text{naive}} = N^2 \times 2N = 2N^3 = 2 \times 1024^3 \approx 2.15 \times 10^9 \]

Tiling 版本：每个 \(T \times T\) 的 tile 被整个线程块共享，每个元素只从全局内存加载一次。沿 K 维需要 \(N/T\) 次 tile 加载，每次加载 \(2 \times T^2\) 个元素：

\[ \text{Accesses}_{\text{tiled}} = \frac{N^2}{T^2} \times \frac{N}{T} \times 2T^2 = \frac{2N^3}{T} = \frac{2 \times 1024^3}{32} \approx 6.71 \times 10^7 \]

减少了 \(T = 32\) 倍！这就是共享内存的巨大价值。

❌ 常见错误：在两次共享内存读写之间忘记调用 __syncthreads()，导致线程读到其他线程尚未写完的旧数据。

✅ 正确模式：加载 tile → __syncthreads() → 计算 → __syncthreads() → 加载下一个 tile。

知识点 4 · GPGPU（通用 GPU 计算）模型

🎯 为什么研究这个问题

GPU 最初专为图形渲染设计，但其大规模并行架构（数千个核心、高内存带宽）使人们意识到：GPU 不仅能画画，更能"算数"。任何可以分解为大量独立并行任务的计算——矩阵运算、图像处理、物理模拟——都可能在 GPU 上获得数量级的加速。

这就是 GPGPU（General-Purpose computing on Graphics Processing Units）的思想。NVIDIA 的 CUDA（2007）和开放标准 OpenCL 为此提供了编程框架，将 GPU 从图形专用设备变成了通用并行计算平台。理解 GPGPU 的线程模型和内存层次，是深度学习、科学计算、高性能计算的基础。

🔍 抽象化方法

CUDA 线程层级（NVIDIA）：

① Grid（网格）：代表整个计算任务，由多个 Block 组成。可以是 1D/2D/3D 结构。

② Block（线程块）：一组可以协作的线程，映射到一个 SM 上执行。块内线程可通过共享内存通信和 __syncthreads() 同步。最大 1024 个线程。

③ Thread（线程）：最小执行单元。每个线程有唯一的 threadIdx（块内）和 blockIdx（网格内）标识。

OpenCL 对应术语：Grid → NDRange，Block → Work-group，Thread → Work-item。概念一一对应。

内存层级（从快到慢）：

① 寄存器（Registers）：每线程私有，最快（0 周期额外延迟），数量有限（~255 个/线程）。

② 共享/本地内存（Shared/Local Memory）：每线程块共享，片上 SRAM，~1-2 周期。

③ 全局内存（Global Memory）：整个 Grid 可见，即 VRAM，~400 周期。

④ 主机内存（Host Memory）：CPU 端 RAM，需通过 PCIe/NVLink 传输，~μs 级延迟。

📐 公式推导 / 模型建立

占用率（Occupancy）是衡量 SM 资源利用率的核心指标：

\[ \text{Occupancy} = \frac{\text{active\_warps}}{\text{max\_warps\_per\_SM}} \]

占用率受三个资源的共同限制：

① 寄存器限制：每线程使用 \(R\) 个寄存器，每块 \(T\) 个线程，则每块需要 \(R \times T\) 个寄存器。SM 有 \(R_{\max}\) 个寄存器，最多容纳 \(\lfloor R_{\max} / (R \times T) \rfloor\) 个块。

② 共享内存限制：每块使用 \(S\) 字节共享内存，SM 有 \(S_{\max}\) 字节，最多 \(\lfloor S_{\max} / S \rfloor\) 个块。

③ 线程数限制：SM 最多支持 \(T_{\max}\) 个线程和 \(B_{\max}\) 个块。

实际块数 = 三个限制中的最小值：

\[ B_{\text{actual}} = \min\!\left(\left\lfloor \frac{R_{\max}}{R \times T}\right\rfloor,\ \left\lfloor \frac{S_{\max}}{S}\right\rfloor,\ \left\lfloor \frac{T_{\max}}{T}\right\rfloor,\ B_{\max}\right) \] \[ \text{Occupancy} = \frac{B_{\text{actual}} \times T / 32}{\text{max\_warps\_per\_SM}} \]
⚠️ 内存合并（Memory Coalescing）是 GPU 全局内存访问的关键优化。当 Warp 中 32 个线程访问连续的内存地址时，硬件将其合并为一次事务（128 字节）。若访问是随机的（非合并），每个线程可能触发独立的事务，浪费 80% 以上的带宽。确保数据布局使相邻线程访问相邻地址！
🌐 实际应用

深度学习：NVIDIA CUDA 生态主导了 AI 训练和推理。cuDNN 库提供高度优化的卷积、GEMM（通用矩阵乘法）实现，PyTorch/TensorFlow 底层调用这些 CUDA 库。GPU 上矩阵乘法可比 CPU 快 100-1000 倍。

科学计算：分子动力学模拟（GROMACS、AMBER）使用 GPU 计算粒子间的力；天气预报模型利用 GPU 加速流体动力学方程求解；基因组学中的序列比对（如 NVIDIA Parabricks）也大量使用 GPU。

图像与视频处理：FFT（快速傅里叶变换）、图像滤波、视频编解码（NVENC/NVDEC）。GPU 的高带宽特别适合像素级并行操作。

加密货币挖矿：早期比特币和以太坊挖矿大量使用 GPU 的并行算力计算哈希函数，推动了 GPU 市场需求（和价格）的飙升。

✏️ 例题分析与易错点

题目：一个 SM 的规格为：64K（65536）个寄存器、48 KB 共享内存、最多 2048 个线程、最多 32 个 Warp。一个内核每线程使用 32 个寄存器，每块 256 个线程，每块使用 8 KB 共享内存。求最大块数和占用率。

解答：

① 寄存器限制：每块需 \(32 \times 256 = 8192\) 个寄存器。\(\lfloor 65536 / 8192 \rfloor = 8\) 个块。

② 共享内存限制：\(\lfloor 49152 / 8192 \rfloor = 6\) 个块（48 KB = 49152 字节）。

③ 线程数限制：\(\lfloor 2048 / 256 \rfloor = 8\) 个块。

取最小值：实际块数 = 6（受共享内存限制）。

活跃线程 = \(6 \times 256 = 1536\)，活跃 Warp = \(1536 / 32 = 48\)。

最大 Warp = \(2048 / 32 = 64\)。

\[ \text{Occupancy} = \frac{48}{64} = 75\% \]

❌ 常见错误：只考虑寄存器或只考虑线程数限制，忽略共享内存的约束。

✅ 正确做法：必须同时检查所有资源限制，取最严格的那个。可使用 NVIDIA Nsight Compute 工具自动分析。

⚡ 快速检测
SM 有 65536 个寄存器，内核每线程用 64 个寄存器、每块 512 个线程。寄存器限制下最多几个块？
