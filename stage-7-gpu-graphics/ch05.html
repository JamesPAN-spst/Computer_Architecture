计算与图像的统一

知识点 1 · Compute Shader 与 CUDA / OpenCL

🎯 为什么研究这个问题

图形管线（Graphics Pipeline）为渲染而生——顶点变换、光栅化、像素着色，一切为画面服务。但 GPU 拥有数千个并行核心和极高的内存带宽，如果只用来画三角形未免太浪费了。

通用计算（GPGPU, General-Purpose computing on GPU）的需求催生了 Compute Shader、CUDA 和 OpenCL——它们让开发者将 GPU 当作大规模并行处理器使用，执行物理模拟、图像处理、机器学习推理等非图形任务。理解这些编程模型，是理解现代 GPU 为何能主导 AI 计算的关键。

🔍 抽象化方法

① Compute Shader

Compute Shader 是图形 API（Vulkan / DirectX 12 / OpenGL）中的通用计算阶段，不参与图形管线的光栅化流程。通过 dispatch(groupsX, groupsY, groupsZ) 启动，每个 work-group 包含固定数量的线程（invocations），线程通过 gl_GlobalInvocationID 等内置变量确定自己的任务。

② CUDA（NVIDIA 专有）

CUDA 使用 kernel<<<grid, block>>>(args) 语法启动内核。Grid 由多个 Block 组成（1D/2D/3D），每个 Block 包含多个线程。Block 内线程可通过 共享内存（Shared Memory）协作。Block 映射到 SM（Streaming Multiprocessor），线程以 Warp（32 线程）为单位调度。

③ OpenCL（跨平台开放标准）

OpenCL 使用 NDRange 定义全局工作空间，划分为 Work-Group，每个包含多个 Work-Item。概念上与 CUDA 的 Grid/Block/Thread 对应，但 OpenCL 可运行在 GPU、CPU、FPGA 等多种设备上。

④ 术语对照

CUDA Block = OpenCL Work-Group = Vulkan Workgroup；CUDA Thread = OpenCL Work-Item = Vulkan Invocation；CUDA Shared Memory = OpenCL Local Memory。

📐 公式推导 / 模型建立

总线程数计算：

\[ N_{\text{threads}} = (G_x \times G_y \times G_z) \times (B_x \times B_y \times B_z) \]

其中 \(G\) 为 grid 维度（block 数量），\(B\) 为 block 维度（线程数量）。

内存合并访问（Coalesced Access）：

同一 Warp 中的 32 个线程若访问连续的 32 × 4 = 128 字节，硬件可合并为一次 128 字节内存事务。非合并访问可能退化为多次事务，带宽利用率大幅下降：

\[ \eta_{\text{BW}} = \frac{\text{有效数据量}}{\text{实际传输量}} \leq 1 \]

Roofline 模型：

定义运算强度 \(I = \dfrac{\text{FLOPs}}{\text{Bytes accessed}}\)，则可达性能为：

\[ P = \min\!\bigl(\,P_{\text{peak}},\; BW_{\text{peak}} \times I\,\bigr) \]

当 \(I < P_{\text{peak}} / BW_{\text{peak}}\)（拐点左侧）时为访存密集型，右侧为计算密集型。

⚠️ 原子操作（Atomic Operations）会序列化线程访问——当大量线程同时对同一地址执行 atomicAdd 时，性能急剧下降。在热路径中应尽量使用分层归约（Hierarchical Reduction）替代全局原子操作。
🌐 实际应用

物理模拟：粒子系统（SPH 流体模拟）、布料仿真——每个粒子/顶点映射到一个线程，利用共享内存处理邻域交互。

图像处理：卷积滤波器（如高斯模糊）、FFT 变换。每个线程处理一个像素，work-group 共享 tile 数据以减少重复加载。

机器学习推理：矩阵乘法（GEMM）是神经网络的核心操作。NVIDIA 提供 cuBLAS（线性代数）、cuDNN（深度学习原语）、TensorRT（推理优化引擎）等高度优化的库。

密码学与科学计算：SHA 哈希并行计算、分子动力学模拟（如 AMBER、GROMACS 的 GPU 加速版本）。

✏️ 例题分析与易错点

题目：设计一个并行归约（Parallel Reduction）求和内核。输入数组长度 \(N = 1{,}048{,}576\)（1M 元素），block 大小 256。描述分层归约策略。

解答：

① 启动 \(N / 256 = 4096\) 个 block，每个 block 处理 256 个元素。

② Block 内归约：每个线程加载一个元素到共享内存。然后执行树形归约（stride 从 128 → 64 → … → 1），每步线程数减半，\(\log_2(256) = 8\) 步后得到 block 级部分和。

③ Block 间归约：每个 block 的部分和通过 atomicAdd 累加到全局结果，或写入中间数组后再启动第二轮归约内核。

\[ \text{Block 内步数} = \log_2(B) = 8, \quad \text{两级总步数} = \log_2(N) = 20 \]

❌ 常见错误：在 block 内归约时忘记调用 __syncthreads()（CUDA）或 barrier()（OpenCL），导致线程间数据竞争。

✅ 正确做法：每步归约之后必须同步，确保前一步写入的结果对所有线程可见。

⚡ 快速检测
某 GPU 峰值计算能力为 10 TFLOPS，显存带宽为 500 GB/s。一个算法的运算强度为 8 FLOPs/Byte，该算法是？

知识点 2 · 光线追踪（Ray Tracing）硬件加速（RT Core）

🎯 为什么研究这个问题

光栅化（Rasterization）是"从物体出发"——将三角形投影到屏幕上，用近似方法模拟光影。它很快，但本质上是伪造光线：反射、折射、全局光照（GI）都需要各种 hack（环境贴图、屏幕空间反射、光照探针……）。

光线追踪（Ray Tracing）是"从眼睛出发"——模拟真实的光线传播物理。每条光线可能与场景中任何物体相交，精确计算反射、折射和阴影。代价是：每条光线需要测试与场景中所有物体的交点——对于百万级三角形的场景，这意味着每帧数十亿次求交运算。没有硬件加速，实时光线追踪根本不可能。

🔍 抽象化方法

① BVH（Bounding Volume Hierarchy）

将场景中的三角形组织为一棵二叉树：每个节点是一个 AABB（Axis-Aligned Bounding Box，轴对齐包围盒），叶节点包含少量三角形。光线先与根节点的 AABB 测试；若未命中则跳过整棵子树，若命中则递归测试子节点。这将 O(n) 的暴力测试降低到 O(log n)。

② RT Core（NVIDIA Turing / Ampere / Ada）

RT Core 是 SM 内的专用硬件单元，加速两个操作：BVH 遍历（Box Intersection）和光线-三角形求交（Triangle Intersection）。当着色器发出 traceRay() 调用时，光线被发送到 RT Core，SM 可以切换到其他 Warp 继续着色——实现计算与求交的流水线并行。

③ DXR / Vulkan Ray Tracing API

DirectX Raytracing（DXR）和 Vulkan Ray Tracing 扩展提供统一的 API：应用程序构建 TLAS（Top-Level Acceleration Structure）和 BLAS（Bottom-Level Acceleration Structure），调用 DispatchRays() 启动光线追踪。

📐 公式推导 / 模型建立

暴力求交 vs BVH：

\[ T_{\text{brute}} = N_{\text{rays}} \times N_{\text{triangles}} \times C_{\text{intersect}} \] \[ T_{\text{BVH}} = N_{\text{rays}} \times O(\log N_{\text{triangles}}) \times C_{\text{intersect}} \]

对于 \(N_{\text{triangles}} = 10^6\)，暴力法需约 \(10^6\) 次测试/光线，BVH 仅需约 \(\log_2(10^6) \approx 20\) 次。加速比约 50,000×。

Möller-Trumbore 光线-三角形求交：

光线方程：\(\mathbf{O} + t\mathbf{D}\)，三角形顶点 \(\mathbf{V}_0, \mathbf{V}_1, \mathbf{V}_2\)。联立得：

\[ \begin{bmatrix} -\mathbf{D} & \mathbf{E}_1 & \mathbf{E}_2 \end{bmatrix} \begin{bmatrix} t \\ u \\ v \end{bmatrix} = \mathbf{T} \]

其中 \(\mathbf{E}_1 = \mathbf{V}_1 - \mathbf{V}_0\)，\(\mathbf{E}_2 = \mathbf{V}_2 - \mathbf{V}_0\)，\(\mathbf{T} = \mathbf{O} - \mathbf{V}_0\)。利用 Cramer 法则求解：

\[ t = \frac{(\mathbf{T} \times \mathbf{E}_1) \cdot \mathbf{E}_2}{(\mathbf{D} \times \mathbf{E}_2) \cdot \mathbf{E}_1}, \quad u = \frac{(\mathbf{D} \times \mathbf{E}_2) \cdot \mathbf{T}}{(\mathbf{D} \times \mathbf{E}_2) \cdot \mathbf{E}_1}, \quad v = \frac{(\mathbf{T} \times \mathbf{E}_1) \cdot \mathbf{D}}{(\mathbf{D} \times \mathbf{E}_2) \cdot \mathbf{E}_1} \]

交点有效条件：\(u \geq 0,\; v \geq 0,\; u + v \leq 1,\; t > 0\)。

⚠️ RT Core 只加速 BVH 遍历和光线-三角形求交——光线命中后的着色计算（材质、光照、纹理采样）仍由常规 SM 核心执行。因此场景复杂着色器（如 PBR 材质 + 多层纹理）仍是性能瓶颈。
🌐 实际应用

混合渲染（Hybrid Rendering）：主可见性（Primary Visibility）使用光栅化（最快），反射、阴影、全局光照使用光线追踪（最准确）。这是当前游戏中最常见的模式。

DXR API 层次：Ray Generation Shader → Intersection/Any-Hit/Closest-Hit Shader → Miss Shader。开发者定义各阶段行为，硬件负责 BVH 遍历。

路径追踪（Path Tracing）：每个像素发射多条光线，模拟光线在场景中的多次弹射（diffuse/specular bounce），最终收敛到真实光照。电影级渲染（如 Pixar RenderMan）采用此方法，每像素可能需要数百到数千条光线。

降噪（Denoising）：路径追踪的噪声问题通过 AI 降噪器（如 NVIDIA OptiX Denoiser）缓解——用较少的光线数量 + AI 降噪获得近似收敛的结果。

✏️ 例题分析与易错点

题目：比较同一游戏场景（百万三角形，1080p）在三种模式下的性能特征：① 纯光栅化 ② 混合光线追踪 ③ 全路径追踪。

解答：

① 纯光栅化：~120 FPS。所有光影通过 shadow map、screen-space reflections (SSR)、baked lightmap 近似。画质有限但速度极快。

② 混合光线追踪：~60 FPS。主可见性光栅化，反射和阴影各发射 1 条光线/像素（~2M 条光线/帧 × 2 = 4M 条）。RT Core 加速求交，着色器处理命中点。画质显著提升（精确反射、柔和阴影）。

③ 全路径追踪：~5-10 FPS（实时）。每像素 8-64 条光线，每条光线多次弹射。2M × 32 × 4 bounces ≈ 2.56 亿条光线/帧。需配合降噪器使用。

❌ 常见错误：认为有了 RT Core 就能实现全路径追踪实时渲染——实际上 RT Core 只加速求交环节，着色和降噪的开销仍然巨大。

✅ 正确认识：当前实时光线追踪的主流方案是混合渲染，全路径追踪仍需搭配 DLSS/降噪等 AI 技术才能接近实时。

知识点 3 · DLSS / FSR——AI 超分辨率

🎯 为什么研究这个问题

分辨率越高，需要着色的像素越多——4K（3840×2160）是 1080p（1920×1080）的4 倍像素，GPU 负担呈平方级增长。传统解决方案是买更强的 GPU——但如果 AI 能从低分辨率图像重建出高分辨率呢？

NVIDIA 的 DLSS（Deep Learning Super Sampling）和 AMD 的 FSR（FidelityFX Super Resolution）正是这一思路的产物。它们让 GPU 以较低分辨率渲染，再通过 AI 或算法升级到目标分辨率，在几乎不损失画质的前提下大幅提升帧率。这项技术正在重新定义"性能 vs 画质"的权衡。

🔍 抽象化方法

① DLSS（Deep Learning Super Sampling, NVIDIA）

DLSS 使用在 NVIDIA 超级计算机上训练的深度神经网络（运行在 Tensor Core 上），结合当前帧的低分辨率渲染、运动向量（Motion Vectors）和前几帧的时域历史（Temporal History）来重建高分辨率图像。本质上是一个时空超分辨率（Temporal Super-Resolution）问题。

② FSR（FidelityFX Super Resolution, AMD）

FSR 1.0：纯空域升级（Spatial Upscaling），使用 Lanczos 滤波 + 边缘检测锐化，不需要运动向量或专用硬件，可在任何 GPU 上运行。FSR 2.0+：加入时域累积（Temporal Accumulation），利用运动向量和多帧信息，画质大幅提升，接近 DLSS。

③ DLSS 3——帧生成（Frame Generation）

DLSS 3 在超分辨率基础上增加了 AI 帧生成：在两个真实帧之间用 Optical Flow Accelerator 生成一帧插值帧。这使得显示帧率可达渲染帧率的 2 倍甚至更高，对 CPU 限制场景尤其有效。

④ 其他竞品

Intel XeSS：在 Intel Arc GPU 上使用 XMX 引擎（类似 Tensor Core），在其他 GPU 上回退到 DP4a 指令。TSR（Temporal Super Resolution）：Unreal Engine 内置的时域升级方案，跨平台通用。

📐 公式推导 / 模型建立

像素数与性能的关系：

\[ \text{性能增益} \approx \frac{\text{目标分辨率像素数}}{\text{渲染分辨率像素数}} = \left(\frac{R_{\text{target}}}{R_{\text{render}}}\right)^2 \]

（假设着色是主要瓶颈，忽略几何处理和升级开销。）

DLSS 各质量模式（4K 输出）：

\[ \begin{aligned} \text{Ultra Quality} &: 2880 \times 1620 \;\Rightarrow\; \frac{3840 \times 2160}{2880 \times 1620} \approx 1.78\times \\ \text{Quality} &: 2560 \times 1440 \;\Rightarrow\; \frac{8{,}294{,}400}{3{,}686{,}400} \approx 2.25\times \\ \text{Balanced} &: 2227 \times 1253 \;\Rightarrow\; \frac{8{,}294{,}400}{2{,}790{,}431} \approx 2.97\times \\ \text{Performance} &: 1920 \times 1080 \;\Rightarrow\; \frac{8{,}294{,}400}{2{,}073{,}600} = 4.0\times \\ \text{Ultra Perf.} &: 1280 \times 720 \;\Rightarrow\; \frac{8{,}294{,}400}{921{,}600} = 9.0\times \end{aligned} \]

实际增益低于理论值，因为超分辨率网络本身有计算开销（约消耗 1-2ms GPU 时间），时域反馈也需要额外显存和带宽。

DLSS Quality 实测典型增益：~2.0-2.5×（理论 2.25×，减去升级开销）。

⚠️ 帧生成（Frame Generation）会增加延迟！生成的插值帧基于历史帧预测，不反映最新的玩家输入。例如玩家在第 N 帧按下鼠标，但生成的第 N+0.5 帧仍显示按下前的状态。NVIDIA Reflex 通过减少渲染队列深度来补偿这一延迟。竞技游戏中应谨慎使用帧生成。
🌐 实际应用

质量模式选择指南：

Ultra Quality / Quality：画质优先，适合单机大作。Quality 是最常用的档位，画质接近原生且帧率提升显著。

Balanced / Performance：帧率优先，适合需要高刷新率的场景。画质有轻微模糊但可接受。

Ultra Performance：极端情况（如 8K 输出），画质损失较大，不推荐在低于 4K 输出时使用。

DLSS 3 帧生成的适用场景：CPU 瓶颈场景（如开放世界游戏中大量 NPC / AI 逻辑）——帧生成在 GPU 端创造额外帧，绕过 CPU 限制。但不适合对输入延迟极其敏感的竞技游戏。

跨平台策略：主机平台（PS5/Xbox）通常使用 TSR 或 FSR，因为它们不依赖特定硬件。PC 平台根据 GPU 品牌选择 DLSS（NVIDIA）或 FSR（AMD/通用）。

✏️ 例题分析与易错点

题目：某游戏原生 4K 渲染帧率为 30 FPS。计算使用 DLSS Quality 和 Performance 模式后的理论帧率及实际预期帧率。

解答：

DLSS Quality（渲染 1440p）：

\[ \text{理论帧率} = 30 \times 2.25 = 67.5 \text{ FPS} \]

扣除升级开销（约 10-15%）：实际 \(\approx 57\text{-}61 \text{ FPS}\)。

DLSS Performance（渲染 1080p）：

\[ \text{理论帧率} = 30 \times 4.0 = 120 \text{ FPS} \]

扣除升级开销：实际 \(\approx 85\text{-}100 \text{ FPS}\)。

❌ 常见错误：直接使用像素比作为帧率乘数。实际中几何处理（顶点着色、曲面细分）不受分辨率影响，且升级网络本身消耗 GPU 资源。

✅ 正确做法：理论增益 × 效率系数（通常 0.7-0.9），具体取决于游戏的着色 vs 几何比例。

知识点 4 · 现代 GPU 的统一着色器架构

🎯 为什么研究这个问题

在 GPU 发展早期（DirectX 9 时代之前），顶点处理和像素处理使用独立的硬件单元——顶点着色器单元（Vertex Shader Units）和像素着色器单元（Pixel Shader Units）是物理上分开的。

这带来一个严重问题：当场景中小三角形密集（像素着色压力大、顶点着色轻松）时，顶点单元闲置；反之（大量复杂几何、简单着色）时，像素单元闲置。无论哪种情况，总有一半硬件在浪费。

统一着色器架构（Unified Shader Architecture）从根本上解决了这一问题——所有着色器类型共享同一组可编程核心，按需动态分配。这一架构变革不仅提升了图形性能，更让 GPU 成为了通用并行计算平台（GPGPU 的硬件基础）。

🔍 抽象化方法

① 固定功能分离架构（Pre-Unified）

NVIDIA GeForce 7 / ATI Radeon X1000 系列：芯片内部有固定数量的专用顶点着色器单元和像素着色器单元。例如 GeForce 7800 GTX 有 8 个顶点单元和 24 个像素单元。硬件比例在芯片设计时固定，无法根据实际工作负载调整。

② 统一着色器架构（Unified Shader Architecture）

NVIDIA GeForce 8800（G80, 2006）和 ATI Radeon HD 2900（R600, 2007）引入统一架构：所有着色器类型（Vertex / Geometry / Pixel / Compute）运行在相同的 SM（Streaming Multiprocessor）或 CU（Compute Unit）上。GPU 调度器动态将 Warp/Wavefront 分配给空闲核心。

③ Mesh Shader（NVIDIA Turing+, 2018）

传统图形管线的顶点 → 曲面细分 → 几何着色器链路固定且效率低下。Mesh Shader 用类似 Compute Shader 的灵活模型取代之：Task Shader（可选，负责剔除和 LOD 决策）→ Mesh Shader（输出 meshlet，即小型三角形组）。GPU 驱动的渲染（GPU-Driven Rendering）成为可能。

📐 公式推导 / 模型建立

固定架构 vs 统一架构的利用率对比：

假设场景中 70% 为像素着色工作量（\(W_P\)），30% 为顶点着色工作量（\(W_V\)），总核心数 \(N\)。

固定架构（假设 75% 核心分配给像素、25% 给顶点）：

\[ T_{\text{fixed}} = \max\!\left(\frac{W_P}{0.75N},\; \frac{W_V}{0.25N}\right) \]

像素时间 = \(\frac{0.7}{0.75N} = \frac{0.933}{N}\)，顶点时间 = \(\frac{0.3}{0.25N} = \frac{1.2}{N}\)

\(T_{\text{fixed}} = \frac{1.2}{N}\)（受限于顶点单元，尽管像素单元仅 78% 负载）。

统一架构（所有核心动态分配）：

\[ T_{\text{unified}} = \frac{W_V + W_P}{N} = \frac{0.3 + 0.7}{N} = \frac{1.0}{N} \]

加速比：

\[ \text{Speedup} = \frac{T_{\text{fixed}}}{T_{\text{unified}}} = \frac{1.2}{1.0} = 1.2\times \]

工作负载越不均衡，统一架构优势越明显。极端情况下（99% 像素、1% 顶点），固定架构的顶点单元严重过载而像素单元几乎空闲。

⚠️ 统一架构并不意味着所有着色器类型的执行效率完全相同——不同着色器对寄存器数量（Register Pressure）和共享内存的需求不同，影响 SM 的 Occupancy（占用率）。复杂的像素着色器可能因寄存器不足导致 Warp 数量减少，降低延迟隐藏能力。
🌐 实际应用

Mesh Shader + GPU-Driven Rendering：传统管线中 CPU 逐物体提交 Draw Call，GPU 被动执行。Mesh Shader 允许 GPU 自主决定渲染内容：Task Shader 在 GPU 上执行视锥剔除（Frustum Culling）和 LOD 选择，Mesh Shader 直接输出可见的 meshlet。CPU 只需提交一次 indirect dispatch。

Nanite（Unreal Engine 5）：虚拟化几何系统，将场景中数十亿三角形组织为 meshlet 层次结构。GPU 动态选择每个物体的适当 LOD 并流式加载，使用 Mesh Shader 或 Compute Shader 实现软件光栅化。开发者无需手动制作 LOD——Nanite 自动处理。

神经渲染（Neural Rendering）：统一架构使得 GPU 可以同时运行图形管线和 AI 推理。NeRF（Neural Radiance Fields）使用神经网络表示 3D 场景，在 GPU 上通过体积渲染 + MLP 推理生成新视角图像。

Compute + Graphics 混合管线：现代游戏引擎大量使用 Compute Shader 实现后处理（Bloom、DOF、TAA）、粒子系统、GPU 剔除等——这些都依赖统一架构的灵活调度能力。

✏️ 例题分析与易错点

题目：某 GPU 有 64 个统一着色器核心。场景 A 的工作负载为 70% 像素着色 + 30% 顶点着色；场景 B 为 40% 像素 + 60% 顶点。比较固定架构（48 像素 + 16 顶点）和统一架构在两个场景中的利用率。

解答：

场景 A（70P / 30V）——固定架构：

\[ T_P = \frac{0.7}{48},\; T_V = \frac{0.3}{16} \;\Rightarrow\; T_P = 0.0146,\; T_V = 0.01875 \]

\(T = \max(0.0146, 0.01875) = 0.01875\)。利用率 = \(\frac{0.7 + 0.3}{64 \times 0.01875} = 83.3\%\)

场景 A——统一架构： \(T = \frac{1.0}{64} = 0.01563\)。利用率 = 100%。

场景 B（40P / 60V）——固定架构：

\[ T_P = \frac{0.4}{48} = 0.00833,\; T_V = \frac{0.6}{16} = 0.0375 \]

\(T = 0.0375\)。利用率 = \(\frac{1.0}{64 \times 0.0375} = 41.7\%\)（顶点严重过载！）

场景 B——统一架构： \(T = \frac{1.0}{64} = 0.01563\)。利用率 = 100%。加速比 = 2.4×。

❌ 常见错误：假设固定架构中两种着色器可以"借用"对方的空闲单元——实际上硬件物理隔离，无法跨类型调度。

✅ 正确理解：统一架构的核心优势就是消除了这种物理隔离，让所有核心成为通用计算资源。
